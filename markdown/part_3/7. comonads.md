## Comonads

Posted by Bartosz Milewski under , ,

This is part 23 of Categories for Programmers. Previously: Monads Categorically. See the Table of Contents.

이것은 프로그래머를 위한 범주론의 23번째 파트입니다. 이전 글: 범주론적 모나드(Monads Categorically). 목차 보기.

Now that we have covered monads, we can reap the benefits of duality and get comonads for free simply by reversing the arrows and working in the opposite category.

이제 우리가 모나드의 험난한 껍질을 전부 까발려 보았으니, 그저 화살표들의 방향이나 좀 휙휙 뒤집고 건너편 쌍대 범주에서 날로 먹는 끄적임만으로도, 쌍대성(duality)의 달콤한 과실을 공짜로 주워 먹으며 코모나드(comonads)라는 녀석을 손쉽게 거머쥘 수 있게 되었습니다.

Recall that, at the most basic level, monads are about composing Kleisli arrows:

단순 무식한 가장 근본적인 차원에서 되짚어보자면, 모나드라는 건 결국 클라이슬리 화살표들을 덕지덕지 합성(composing)해 나가는 짓거리였음을 부디 기억하시기 바랍니다:

```
a -> m b
```

where m is a functor that is a monad. If we use the letter w (upside down m) for the comonad, we can define co-Kleisli arrows as morphism of the type:

여기서 `m`은 모나드의 탈을 쓴 영악한 함자입니다. 만약 우리가 코모나드(comonad)를 상징하는 징표로 `w` (`m`을 우스꽝스럽게 거꾸로 뒤집은 모양새죠)라는 글자를 갖다 쓴다면, 앞서와의 정반대로 코-클라이슬리 화살표(co-Kleisli arrows)를 이딴 형태의 사상으로 버젓이 정의해 낼 수 있게 됩니다:

```
w a -> b
```

The analog of the fish operator for co-Kleisli arrows is defined as:

코-클라이슬리 화살표를 이어주기 위해 고안된 물고기 연산자(fish operator)의 유사품은 다음과 같이 정의 내려집니다:

```
(=>=) :: (w a -> b) -> (w b -> c) -> (w a -> c)
```

For co-Kleisli arrows to form a category we also have to have an identity co-Kleisli arrow, which is called extract:

이 알량한 코-클라이슬리 화살표 무리들이 당당히 하나의 범주를 형성해 내려면, 당연하게도 그들 사이를 중재할 항등 코-클라이슬리 화살표 역시 반드시 하나쯤 끼고 있어야만 하는데, 이 녀석의 이름은 바로 추출(`extract`)이라 불립니다:

```
extract :: w a -> a
```

This is the dual of return. We also have to impose the laws of associativity as well as left- and right-identity. Putting it all together, we could define a comonad in Haskell as:

이것이야말로 그토록 징글맞던 `return`의 쌍대(dual)입니다. 우리는 또한 예외 없이 결합 법칙을 비롯해 왼쪽과 오른쪽의 항등 법칙 따위도 구속력 있게 강제(impose)해야만 합니다. 이 모든 자질구레한 파편들을 한데 끌어모으면, 우리는 마침내 Haskell에서 코모나드를 이와 같이 정의해 낼 수 있습니다:

```
class Functor w => Comonad w where
    (=>=) :: (w a -> b) -> (w b -> c) -> (w a -> c)
    extract :: w a -> a
```

In practice, we use slightly different primitives, as we’ll see shortly.

실제 프로그래밍의 수렁에선 우리가 곧 눈으로 확인하게 될 테지만, 이와는 아주 살짝 빗나간 형태의 원시 연산들(primitives)을 사용하곤 합니다.

The question is, what’s the use for comonads in programming?

여기서 드는 의문은 시발 이거, 도대체 코모나드 따위를 프로그래밍판에서 어디다 써먹냐는 겁니다.

## Programming with Comonads

## 코모나드와 프로그래밍(Programming with Comonads)

Let’s compare the monad with the comonad. A monad provides a way of putting a value in a container using return. It doesn’t give you access to a value or values stored inside. Of course, data structures that implement monads might provide access to their contents, but that’s considered a bonus. There is no common interface for extracting values from a monad. And we’ve seen the example of the IO monad that prides itself in never exposing its contents.

모나드와 코모나드를 샅샅이 비교 대조해 봅시다. 한 마리의 모나드라는 건 `return`을 도구 삼아 껍데기(컨테이너) 안에 어떤 알맹이 값을 불쑥 쑤셔 넣는 수법을 제공합니다. 하지만 그 속에 욱여넣어진 가련한 값들에 도로 접근할 길 따위는 좀처럼 열어주지 않죠. 물론 모나드를 흉내 내어 구현한 잡다한 데이터 구조들이야 각기 제 나름의 방식으로 알맹이를 꺼낼 구멍을 파두었을지는 몰라도, 그건 그저 운 좋은 덤(bonus)으로 치부될 뿐입니다. 어떤 모나드에서든 공통으로 통용될 만하게, 알맹이 값들을 쑥 뽑아내는 범용 인터페이스(common interface) 같은 건 이 세상에 존재하지 않습니다. 당장 그 콧대 높은 `IO` 모나드 나부랭이만 보더라도, 속내를 절대 바깥에 까발리지 않는 그 징그러운 폐쇄성을 자기만의 알량한 자부심거리로 삼아대던 꼴을 우리는 이미 두 눈으로 똑똑히 목격했지 않습니까.

A comonad, on the other hand, provides the means of extracting a single value from it. It does not give the means to insert values. So if you want to think of a comonad as a container, it always comes pre-filled with contents, and it lets you peek at it.

이와 정반대로, 코모나드라는 녀석은 바로 그 속에서 단 하나의 값 조각을 날름 끄집어내는 수단만을 기가 막히게 제공해 줍니다. 헌데 정작 거기다 무언가를 새로 욱여넣을 방도라곤 한 구석에도 열어두지 않습니다. 고로 만약 당신이 이 코모나드라는 기이한 물건을 일종의 컨테이너로 받아들이고자 한다면, 이것은 시초부터 꽉꽉 들어찬 내용물들로 항상 선탑재(pre-filled)된 채 툭 던져지며, 오로지 여러분이 그 안을 슬쩍 훔쳐볼(peek) 수 있게만 허락해 주는 요지경 같은 존재라 여겨야만 합니다.

Just as a Kleisli arrow takes a value and produces some embellished result — it embellishes it with context — a co-Kleisli arrow takes a value together with a whole context and produces a result. It’s an embodiment of contextual computation.

일찍이 클라이슬리 화살표가 그저 헐벗은 값 하나를 받아다가 거기에 모종의 꾸밈(문맥)을 덕지덕지 발라 치장된 결과물로 뱉어냈던 것처럼 — 코-클라이슬리 화살표는 반대로 아예 문맥(context) 덩어리를 통째로 뒤집어쓴 값 자체를 한 번에 꿀꺽 삼킨 뒤, 그냥 깔끔한 알맹이 결과물 하나를 달랑 뱉어냅니다. 이는 말하자면 문맥적 연산(contextual computation)의 완벽한 화신입니다.

## The Product Comonad

## 코프러덕트 코모나드: 곱 코모나드(The Product Comonad)

Remember the reader monad? We introduced it to tackle the problem of implementing computations that need access to some read-only environment e. Such computations can be represented as pure functions of the form:

예전의 그 리더 모나드(reader monad)를 아직도 잊지 않으셨나요? 우리는 그것을, `e`라는 모종의 읽기 전용 환경(read-only environment)에 접속해 얌전히 눈팅이나 해야만 굴러가는 연산들을 구현해 낼 때 쏠쏠하게 수단으로 써먹으려고 세상에 첫선을 보였었죠. 그런 시시콜콜한 연산들 따위는 아래와 같은 뻔한 형색을 띤 순수 함수들로 너끈히 표현될 수 있었습니다:

```
(a, e) -> b
```

We used currying to turn them into Kleisli arrows:

그리고 우린 이 가련한 녀석들을 커링(currying)이라는 도마 위에 올려 난도질함으로써 기어코 클라이슬리 화살표의 면상으로 둔갑시켜 버렸습니다:

```
a -> (e -> b)
```

But notice that these functions already have the form of co-Kleisli arrows. Let’s massage their arguments into the more convenient functor form:

그런데 가만히 한 번 노려보시죠. 이 원초적인 함수들의 꼬락서니가 이미 처음부터 영락없는 저 코-클라이슬리 화살표 꼴을 다 갖추고 있었단 사실을 눈치채셨습니까? 자, 이 인수의 덩어리들을 우리가 주무르기 좀 더 편하게스리 어엿한 함자의 탈을 쓰도록 부드럽게 마사지(massage)해 봅시다:

```
data Product e a = Prod e a
  deriving Functor
```

We can easily define the composition operator by making the same environment available to the arrows that we are composing:

우린 그저 우리가 한데 엮으려는 저 화살표 놈들한테 매번 똑같은 환경(environment) 값을 공평하게 내어주기로 작정하기만 하면, 속 편하고 손쉽게 이 합성 연산자를 떡하니 정의해 낼 수 있습니다:

```
(=>=) :: (Product e a -> b) -> (Product e b -> c) -> (Product e a -> c)
f =>= g = \(Prod e a) -> let b = f (Prod e a)
                          c = g (Prod e b)
                       in c
```

The implementation of extract simply ignores the environment:

저 추출(`extract`) 함수를 구현하는 건 기막히게 허탈할 지경입니다. 그저 그 환경 따위를 개무시해버리면 그만이니까요:

```
extract (Prod e a) = a
```

Not surprisingly, the product comonad can be used to perform exactly the same computations as the reader monad. In a way, the comonadic implementation of the environment is more natural — it follows the spirit of “computation in context.” On the other hand, monads come with the convenient syntactic sugar of the do notation.

하등 놀라울 것도 없이, 이 곱 코모나드(product comonad)라는 녀석은 리더 모나드가 여지껏 뺑이치던 것과 털끝만치도 다를 바 없이 똑같은 연산들을 척척 수행해 내는 데에 안성맞춤으로 돌려막기 될 수 있습니다. 어떤 시각으로 뜯어보면, 환경(environment)을 코모나드의 껍질로 포장해 구현해 낸 꼬락서니가 사뭇 더 자연스러워 보이기도 합니다 — 기실 이것은 "저 문맥 속에서의 연산(computation in context)"이라는 코모나드의 우매한 강령(spirit)을 소름 끼치도록 가장 충실하게 받들어 모시는 꼬라지니 말입니다. 허나 반대편의 모나드 쪽 진영엔 `do` 표기법이라는 악마같이 달콤하고 편리한 문법적 설탕(syntactic sugar)의 유혹이 언제나 도사리고 있습니다.

The connection between the reader monad and the product comonad goes deeper, having to do with the fact that the reader functor is the right adjoint of the product functor. In general, though, comonads cover different notions of computation than monads. We’ll see more examples later.

저 리더 모나드와 이 곱 코모나드 간의 구질구질한 밀월 관계의 뿌리는 실로 더 아득히 깊은 수렁으로 파고들어가는데, 이는 저 리더 함자라는 놈이 다름 아닌 곱 함자의 우쪽 수반(right adjoint) 노릇을 꿰차고 있다는 경악스러운 진실과 질척하게 뒤얽혀 있습니다. 그렇긴 해도 좀 더 구름 위 차원에서 뭉뚱그려 말하자면, 코모나드 무리는 대게 모나드 것들과는 궤를 완전히 달리하는 별개의 연산 개념(notions of computation)들을 아우르며 포섭합니다. 이는 후에 다른 예시들에서 더 낱낱이 파헤쳐보게 될 것입니다.

It’s easy to generalize the Product comonad to arbitrary product types including tuples and records.

그저 튜플(tuples)이나 레코드(records) 따위를 아우르는 하찮은 임의의 곱 타입(product types)들에까지 이 곱 코모나드의 성질을 스멀스멀 확장시켜 일반화(generalize)하는 일은 한낱 어린아이 장난에 불과합니다.

## Dissecting the Composition

## 합성의 해부(Dissecting the Composition)

Continuing the process of dualization, we could go ahead and dualize monadic bind and join. Alternatively, we can repeat the process we used with monads, where we studied the anatomy of the fish operator. This approach seems more enlightening.

끝나지 않는 쌍대화(dualization) 작업의 연장선상에서, 우리는 내친김에 모나드의 바인드(`bind`)와 조인(`join`)마저도 거침없이 쌍대화해버릴 수 있습니다. 하지만 그 대신, 예전에 우리가 모나드를 상대하며 써먹었던 그 방식 — 바로 물고기 연산자의 해부학적 얼개를 샅샅이 파고들었던 그 과정을 여기서 다시 똑같이 되풀이해볼 수도 있습니다. 이 접근법이 필경 훨씬 더 밝고 탁 트인 깨달음을 선사해줄 듯싶습니다.

The starting point is the realization that the composition operator must produce a co-Kleisli arrow that takes w a and produces a c. The only way to produce a c is to apply the second function to an argument of the type w b:

시작점은, 이 합성 연산자라는 녀석이 응당 `w a`를 받아 `c`를 굽어 뽑아내는 꼴의 코-클라이슬리 화살표를 산출해야만 한다는 뼈저린 자각입니다. 하지만 `c`를 창조해 낼 유일무이한 방도란 오직 두 번째 함수에다 `w b` 타입의 인수를 질러 넣고 적용시키는 것뿐입니다:

```
(=>=) :: (w a -> b) -> (w b -> c) -> (w a -> c)
f =>= g = g ...
```

But how can we produce a value of type w b that could be fed to g? We have at our disposal the argument of type w a and the function f :: w a -> b. The solution is to define the dual of bind, which is called extend:

허나 대체 무슨 수로 저 `g`놈의 아가리에 처박힐 `w b` 타입의 값을 우리가 맨손으로 연성해 낼 수 있단 말입니까? 지금 우리 손모가지에 쥐어진 패라곤 고작 `w a` 타입의 인수 쪼가리 하나와, `f :: w a -> b`라는 알량한 함수 하나가 전부입니다. 이 막막한 사태를 타개할 유일한 탈출구는 바로 바인드(`bind`)의 한 핏줄 쌍대격인, 이른바 연장(`extend`)이라 불리는 자를 새롭게 소환하여 정의하는 길뿐입니다:

```
extend :: (w a -> b) -> w a -> w b
```

Using extend we can implement composition:

이 연장(`extend`) 기능을 동원하면 마침내 합성을 저지르고 구현해 낼 수 있습니다:

```
f =>= g = g . extend f
```

Can we next dissect extend? You might be tempted to say, why not just apply the function w a -> b to the argument w a, but then you quickly realize that you’d have no way of converting the resulting b to w b. Remember, the comonad provides no means of lifting values. At this point, in the analogous construction for monads, we used fmap. The only way we could use fmap here would be if we had something of the type w (w a) at our disposal. If we coud only turn w a into w (w a). And, conveniently, that would be exactly the dual of join. We call it duplicate:

그럼 이번엔 저 연장(`extend`) 녀석의 배를 가르고 해부해 볼까요? 아마 여러분은 무심히 "야, 그냥 `w a -> b` 함수에다 대충 `w a` 인수 때려 박고 치우면 안 돼?"라고 속편하게 지껄이고 싶은 유혹에 시달리겠지만, 금세 당신은 그 허접한 결과물로 튀어나온 `b`를 어떻게 다시 `w b`로 포장해 바꿀지 도무지 갈피조차 잡을 수 없다는 비참한 통찰에 도달하게 될 겁니다. 기억해 두십시오. 코모나드라는 놈은 값 따위를 함부로 들어올릴(`lift`) 수단 따윈 한 줌도 허락하지 않습니다. 여지껏 모나드의 구역에서 비슷하게 굴러먹던 짓거리에선 우리가 뻔질나게 `fmap`을 우려먹었죠. 헌데 여기서도 우리가 그 빌어먹을 `fmap`을 한 번 써먹으려 든다면, 우리 수중에 오롯이 `w (w a)` 타입의 무언가가 자리 잡고 있어야만 그나마 말이 됩니다. 만약 우리가 어떻게든 저 비루한 `w a`를 쥐어짜 내어 억지로 `w (w a)`의 탈을 뒤집어씌울 수만 있다면 말입니다. 그리고 참으로 경악스럽게도 기가 막힌 우연이겠지만, 그 행위야말로 정확히 조인(`join`)의 쌍대(dual)에 해당하는 짓거리입니다. 우린 이 흉악한 놈을 복제(`duplicate`)라고 부를 겁니다:

```
duplicate :: w a -> w (w a)
```

So, just like with the definitions of the monad, we have three equivalent definitions of the comonad: using co-Kleisli arrows, extend, or duplicate. Here’s the Haskell definition taken directly from Control.Comonad library:

그리하여 모나드가 가졌던 수많은 정의들과 소름 돋게 똑같이 판박이처럼, 코모나드 역시 서로가 필연적으로 동치(equivalent)인 3가지의 정의를 품에 안게 되었습니다: 바로 코-클라이슬리 화살표를 써먹거나, 아니면 연장(`extend`), 그도 아니면 복제(`duplicate`)를 부리는 방식들이죠. 저기 `Control.Comonad` 라이브러리에서 대놓고 통째로 뜯어온 Haskell의 적나라한 정의를 감상하시죠:

```
class Functor w => Comonad w where
  extract :: w a -> a
  duplicate :: w a -> w (w a)
  duplicate = extend id
  extend :: (w a -> b) -> w a -> w b
  extend f = fmap f . duplicate
```

Provided are the default implementations of extend in terms of duplicate and vice versa, so you only need to override one of them.

여기엔 그저 `duplicate`를 볼모 삼아 연성해 낸 `extend`의 기본 구현과 아울러, 반대로 `extend`를 베끼고 등쳐먹어 짜 맞춘 `duplicate`의 기본 구현이 상호 교차로 제공되기 때문에, 거저먹고 싶다면 그저 당신 꼴리는 대로 둘 중 단 하나만 구미에 맞게 덮어써(override) 버리면 만사 장땡입니다.

The intuition behind these functions is based on the idea that, in general, a comonad can be thought of as a container filled with values of type a (the product comonad was a special case of just one value). There is a notion of the “current” value, one that’s easily accessible through extract. A co-Kleisli arrow performs some computation that is focused on the current value, but it has access to all the surrounding values. Think of the Conway’s game of life. Each cell contains a value (usually just True or False). A comonad corresponding to the game of life would be a grid of cells focused on the “current” cell.

이 괴상망측한 함수들의 뒷골목에 도사리고 있는 고상한 직관(intuition)이란 건 사실, 통념상 코모나드란 놈을 `a` 타입의 숱한 알맹이들로 그득그득 속이 꽉꽉 들어차 터질 듯한 하나의 큰 용기(container) 따위로 여겨도 일체 무방하다는 착상에 깊게 뿌리내리고 있습니다(아까 그 곱 코모나드라는 건 그저 알맹이가 달랑 하나뿐인 아주 빈약하고 특수한 예외 케이스였을 뿐이죠). 여하튼 이 용기 안에는 지금 현재 내가 주시하며 물고 있는 이른바 "현재(current)" 값이라는 요상한 개념이 자리 잡고 있는데, 이것이야말로 추출(`extract`)을 써서 아주 만만하게 쏙 뽑아먹을 수 있는 녀석입니다. 코-클라이슬리 화살표라는 건 다름 아니라 오로지 이 불운한 "현재" 값을 집중적으로 패대는 연산을 갈기면서도, 아주 이기적이게도 그 주변을 둘러싼 여타 모든 알맹이 값들엔 뻔뻔하게 접근 권한을 행사하는 폭군 같은 작성을 지니고 있습니다. 그 유명한 콘웨이의 생명 게임(Conway's game of life)을 한번 떠올려 보십시오. 각 세포(cell)는 어떤 특정한 값(보통 `True` 아니면 `False`)을 머금고 있습니다. 이 미친 생명 게임에 완벽히 호응하는 코모나드란 건, 결국 저 숱하게 널려있는 세포들의 격자판 자체이며, 오직 지금 관찰의 사정거리에 놓인 "현재"의 세포 하나에만 온 시선이 잔뜩 몰려 초점이 잡혀 있는 섬뜩한 형국인 셈입니다.

So what does duplicate do? It takes a comonadic container w a and produces a container of containers w (w a). The idea is that each of these containers is focused on a different a inside w a. In the game of life, you would get a grid of grids, each cell of the outer grid containing an inner grid that’s focused on a different cell.

그렇다면 과연 도대체 복제(`duplicate`)란 놈은 무슨 망나니 짓을 벌이는 걸까요? 이 녀석은 `w a`라는 코모나드 용기 하나를 날름 뺏어 먹더니, 미치광이 짓거리를 부려 돌연 또 다른 용기들을 담고 있는 더 악랄한 대형 용기 `w (w a)` 하나를 세상에 토스하여 뱉어놓습니다. 굳이 이 사태를 납득해 보자면, 이 알량한 하위 용기들 하나하나는 결국 본래의 코모나드 `w a` 내부의 각기 다른 숱한 `a` 요소들 각각에게 일일이 집요하게 초점이 맞춰진(focused on) 감시 체계란 소리입니다. 생명 게임에 비유하자면, 그건 격자판을 가둬두는 또 다른 격자판을 하나 사은품으로 얻는 격입니다. 그 거대하고 불길한 외부 격자판의 낱낱의 세포마다, 정작 안에선 원래 생태계의 각기 다른 세포 하나하나를 집요하게 조준하고 있는 내부 격자판의 환영이 통째로 처박혀 옥살이를 하고 있는 모양새라 할 수 있죠.

Now look at extend. It takes a co-Kleisli arrow and a comonadic container w a filled with as. It applies the computation to all of these as, replacing them with bs. The result is a comonadic container filled with bs. extend does it by shifting the focus from one a to another and applying the co-Kleisli arrow to each of them in turn. In the game of life, the co-Kleisli arrow would calculate the new state of the current cell. To do that, it would look at its context — presumably its nearest neighbors. The default implementation of extend illustrates this process. First we call duplicate to produce all possible foci and then we apply f to each of them.

이제 연장(`extend`)을 한 번 노려봅시다. 이 교활한 녀석은 코-클라이슬리 화살표 하나, 그리고 `a` 잡동사니들로 그득그득 터져 나가는 코모나드의 용기 `w a`를 한 쌍으로 채갑니다. 그러고는 그 무지몽매한 수많은 `a` 놈들에게 무자비한 연산의 철퇴를 일괄적으로 내리꽂아, 기어이 전부 다 `b`라는 다른 족속들로 싹 다 갈아치워 치환해버립니다. 그 대학살의 결과로 떡하니 남는 건, 어느새 남의 집이던 `b` 무리들로 득시글득시글 꽉 차버린 또 하나의 코모나드 용기뿐입니다. 그런데 이 `extend`가 그 흉악무도한 짓거리를 어떻게 그토록 섬세하게 수행하느냐면, 녀석은 그예 사냥감의 초점(focus)을 어떤 불운한 `a` 놈에게서 또 다른 가련한 놈에게로 끊임없이 옮겨 다녀가며 차례차례 그 무시무시한 코-클라이슬리 화살표의 율법을 공평히 집행하는 수법을 구사합니다. 앞선 생명 게임에 대조해보면, 코-클라이슬리 화살표가 하는 짓거리란 뻔하게도 지금 초점이 박힌 "현재" 세포의 새로운 운명(상태)을 재판하여 선고하는 짓입니다. 헌데 그 음험한 작태를 도모한답시고 녀석은 뻔질나게 주위의 문맥(context) — 필시 가장 인접한 이웃 녀석들의 동태겠죠 — 을 음흉하게 훔쳐보고 눈치를 살피곤 할 겁니다. `extend`의 기본 구현 코드가 바로 이 역겨울 만큼 정교하고 기계적인 과정을 한 편의 해부도처럼 적나라하게 전시합니다: 우선은 미친 듯이 `duplicate`를 남발하여 모든 세포에 각기 초점이 아로새겨진 환영의 감옥들(가능한 모든 초점들)을 징그럽게 양산해 낸 뒤, 그 포로수용소의 각 객체마다 하나하나 자비 없이 저 함수 `f`를 때려 박아버리는 식이죠.

## The Stream Comonad

## 스트림 코모나드(The Stream Comonad)

This process of shifting the focus from one element of the container to another is best illustrated with the example of an infinite stream. Such a stream is just like a list, except that it doesn’t have the empty constructor:

용기 안에 든 이놈의 요소에서 저놈의 요소로 초점(focus)을 줄기차게 옮겨대는 이 끔찍한 과정살이는, 다름 아닌 무한 스트림(infinite stream)의 사례를 끌고 왔을 때 그 진면목이 가장 적나라하게 폭로됩니다. 그러한 스트림은 그저 빈 생성자(empty constructor) 따위의 얄궂은 도피처 하나가 쏙 빠져있다는 점만 빼면 어차피 한낱 평범한 리스트 쪼가리와 하등 다를 바가 없습니다:

```
data Stream a = Cons a (Stream a)
```

It’s trivially a Functor:

이 따위 것은 시시하게도 너무나 당연한 함자(Functor)입니다:

```
instance Functor Stream where
    fmap f (Cons a as) = Cons (f a) (fmap f as)
```

The focus of a stream is its first element, so here’s the implementation of extract:

어떤 스트림의 초점(focus)이란 볼 것도 없이 그 첫 번째 대갈통(요소)에 꽂혀 있으니, 당연하게도 여기서의 추출(`extract`) 구현은 이따위로 짜여집니다:

```
extract (Cons a _) = a
```

duplicate produces a stream of streams, each focused on a different element.

가련한 복제(`duplicate`)는 각기 다른 요소들에 기괴하게 초점이 박힌 스트림들의 스트림 나부랭이를 꾸역꾸역 생산해 냅니다.

```
duplicate (Cons a as) = Cons (Cons a as) (duplicate as)
```

The first element is the original stream, the second element is the tail of the original stream, the third element is its tail, and so on, ad infinitum.

맨 첫 번째 덩어리는 원판 스트림 그 자체고, 두 번째 놈은 그 원판의 꼬리(tail)를 잘라낸 거고, 세 번째 녀석은 또다시 그놈의 꼬리며, 이 끔찍한 족보는 끝도 없이 꼬리를 물고 무한히(ad infinitum) 이어집니다.

Here’s the complete instance:

자, 여기 토악질 나오는 완성본 인스턴스 납십니다:

```
instance Comonad Stream where
    extract (Cons a _) = a
    duplicate (Cons a as) = Cons (Cons a as) (duplicate as)
```

This is a very functional way of looking at streams. In an imperative language, we would probably start with a method advance that shifts the stream by one position. Here, duplicate produces all shifted streams in one fell swoop. Haskell’s laziness makes this possible and even desirable. Of course, to make a Stream practical, we would also implement the analog of advance:

이거야말로 무한의 스트림을 철저히 다분히 함수형다운 작태로 뜯어보는 방식입니다. 명시적인 명령형 언어 골목에선 아마 십중팔구 스트림 자리를 딸깍 한 칸 밀어내는 `advance` 따위의 메서드나 주물럭거리며 시작을 끊었겠죠. 허나 여기선 그 지독한 `duplicate`가 그 모든 쓸려 나간 스트림의 전생과 후생을 단 한 방(in one fell swoop)에 통째로 뽑아내 버립니다. 오로지 Haskell의 끔찍한 지연 평가(laziness)만이 이 미친 짓을 가능케 삼, 심지어 부추기는 원동력이 됩니다. 뭐 당연하겠지만, 이 `Stream`이란 걸 현실에서 써먹으려면 우리 역시 그 추잡한 `advance` 비스무리한 아류작 하나쯤은 덤으로 구현해야만 직성이 풀리긴 합니다:

```
tail :: Stream a -> Stream a
tail (Cons a as) = as
```

but it’s never part of the comonadic interface.

그치만 그건 기실 단 한 번도 신성한 코모나드 인터페이스의 정식 구역에 끼어든 적 없는 잡종 나부랭이일 뿐입니다.

If you had any experience with digital signal processing, you’ll see immediately that a co-Kleisli arrow for a stream is just a digital filter, and extend produces a filtered stream.

행여 디지털 신호 처리(DSP)판에서 굴러먹어 본 짬밥이 있는 자라면, 이 한낱 스트림을 향유하는 코-클라이슬리 화살표 따위가 그저 시시한 디지털 필터(digital filter) 쪼가리에 불과하며, 저 빌어먹을 수의 연장(`extend`)이 단지 필터링 된 찌꺼기 스트림 하나를 내뱉아 준다는 사실쯤은 단박에 식별해 낼 수 있을 겁니다.

As a simple example, let’s implement the moving average filter. Here’s a function that sums n elements of a stream:

단순 무식한 끄나풀로, 당장 이동 평균 필터(moving average filter)나 하나 대충 찍어내 봅시다. 자, 여기 어떤 스트림의 `n`개 분량의 모가지를 죄다 틀어쥐고 합산해 내는 함수 대령입니다:

```
sumS :: Num a => Int -> Stream a -> a
sumS n (Cons a as) = if n <= 0 then 0 else a + sumS (n - 1) as
```

Here’s the function that calculates the average of the first n elements of the stream:

그리고 이건 그 스트림의 첫 대가리 `n`개 요소들의 평균값을 후려치는 함수입니다:

```
average :: Fractional a => Int -> Stream a -> a
average n stm = (sumS n stm) / (fromIntegral n)
```

Partially applied average n is a co-Kleisli arrow, so we can extend it over the whole stream:

부분 적용(Partially applied)의 형벌을 받은 `average n`은 마침내 그럴싸한 코-클라이슬리 화살표의 꼬락서니를 갖췄으니, 우린 이걸 전체 스트림의 뼈대에 쭈욱 펴 바를(`extend`) 수 있게 됩니다:

```
movingAvg :: Fractional a => Int -> Stream a -> Stream a
movingAvg n = extend (average n)
```

The result is the stream of running averages.

그 대학살의 귀결로, 숨 가쁘게 뒤쫓는 누적 평균(running averages)들의 기나긴 스트림이 징그럽게 토해집니다.

A stream is an example of a unidirectional, one-dimensional comonad. It can be easily made bidirectional or extended to two or more dimensions.

이 스트림이라는 건 한낱 단방향(unidirectional)에 그치는 1차원적인 시시한 코모나드의 찌꺼기에 다름없습니다. 마음만 먹으면 이건 손쉽게 쌍방향(bidirectional)으로 뒤집어 파내거나 2차원, 3차원으로 미친 듯이 확장(`extended`)해 나갈 수도 있습니다.

## Comonad Categorically

## 범주론적 코모나드(Comonad Categorically)

Defining a comonad in category theory is a straightforward exercise in duality. As with the monad, we start with an endofunctor T. The two natural transformations, η and μ, that define the monad are simply reversed for the comonad:

범주론의 세계에서 코모나드를 정의해 내는 일은 그야말로 쌍대성(duality)을 다루는 단순무식하고 직관적인 노가다에 불과합니다. 모나드를 다룰 때와 똑같이 우린 또다시 자기 함자 `T` 하나를 집어 들고 시작합니다. 모나드를 규정하던 두 개의 자연 변환, 즉 저 `η`와 `μ`는 이 코모나드 구역에 와선 그저 뒤짱구처럼 휙 뒤집히기만 하면 그만입니다:

```
ε :: T -> I
δ :: T -> T2
```

The components of these transformations correspond to extract and duplicate. Comonad laws are the mirror image of monad laws. No big surprise here.

이 알량한 변환들의 각기 속한 성분(components)들은 고스란히 저 추출(`extract`)과 복제(`duplicate`)의 쌍판과 정면으로 맞대응합니다. 코모나드의 율법이란 것도 기껏해야 모나드 법칙들을 거울에 비춘 꼴인 좌우 반전에 지나지 않습니다. 여기엔 그 어떤 되먹지 못한 큰 반전도, 놀라움도 없습니다.

Then there is the derivation of the monad from an adjunction. Duality reverses an adjunction: the left adjoint becomes the right adjoint and vice versa. And, since the composition R ∘ L defines a monad, L ∘ R must define a comonad. The counit of the adjunction:

그다음에 들이닥치는 건 예전 수반(adjunction)에서 모나드를 쥐어짜 내던 짓거리의 복제판입니다. 쌍대성이란 무자비놈은 수반마저 뒤엎어버립니다: 왼쪽 수반(left adjoint)은 어김없이 오른쪽 수반이 돼버리고, 반대 방향도 마찬가지 꼴을 당하죠. 그리고 기왕지사 저 합체의 찌꺼기인 `R ∘ L`이 모나드를 찍어냈었으니, 이쪽 짝수반 찌꺼기인 `L ∘ R`이 필연적으로 코모나드를 찍어내야만 하는 건 자명한 이치입니다. 그 수반의 여단일원(counit) 녀석 말입니다:

```
ε :: L ∘ R -> I
```

is indeed the same ε that we see in the definition of the comonad — or, in components, as Haskell’s extract. We can also use the unit of the adjunction:

이 녀석은 진실로 우리가 조금 전에 코모나드의 정의에서 얼핏 훔쳐봤던 그 `ε` 놈과 일점일획도 틀림없이 동일한 판박이이며 — 성분의 관점에서 파보면 그게 바로 Haskell의 뻔질난 `extract`입니다. 우린 또한 기꺼이 수반의 단일원(unit) 역시 도스름하게 써먹을 수 있습니다:

```
η :: I -> R ∘ L
```

to insert an R ∘ L in the middle of L ∘ R and produce L ∘ R ∘ L ∘ R. Making T2 from T defines the δ, and that completes the definition of the comonad.

저 단일원을 써먹어 `L ∘ R`의 정중앙 배를 가르고 그 안에 `R ∘ L`을 불쑥 구겨 넣어, 이내 끔찍한 혼종 `L ∘ R ∘ L ∘ R`을 뱉어내 버리는 짓거리에 말입니다. 그렇게 `T` 하나를 쥐어뜯어 `T2`로 부풀려 내는 이 만행이 바로 저 `δ`의 실체를 정의해 내며, 이로써 코모나드를 향한 지지부진한 정의가 마침내 완전한 종지부를 찍습니다.

We’ve also seen that the monad is a monoid. The dual of this statement statement would require the use of a comonoid, so what’s a comonoid? The original definition of a monoid as a single-object category doesn’t dualize to anything interesting. When you reverse the direction of all endomorphisms, you get another monoid. Recall, however, that in our approach to a monad, we used a more general definition of a monoid as an object in a monoidal category. The construction was based on two morphisms:

우린 일찍이 모나드란 게 까고 보면 그저 모노이드 나부랭이에 불과하단 사실을 깨달았습니다. 그렇다면 이 참으로 얼탱이 없는 선언의 쌍대(dual) 격 진술이란 건 필경 코모노이드(comonoid)란 무언가를 소환해야 할 텐데, 대체 이 코모노이드란 건 또 무슨 근본 없는 녀석일까요? 모노이드를 고작 단일 대상 범주(single-object category)로 날름 퉁쳤던 원초적인 정의는, 이를 아무리 뼛속까지 쌍대화시켜 봤자 단 1의 흥미로운 결과물도 내놓지 못합니다. 당신이 그놈의 자기 사상(endomorphisms) 화살표들의 모가지를 싹 다 거꾸로 꺾어 돌려놔 본댔자, 결국 기어 나오는 건 또 다른 모노이드 한 마리일 뿐이거든요. 하지만 우린 예전 모나드를 요리할 때, 모노이드를 그저 모노이드 범주(monoidal category) 안에 기거하는 어엿한 대상(object)이라고 한 차원 더 잘난 척하며 일반화시켰던 전적을 부디 환기해 보십시오. 그 거창한 건축물은 달랑 두 개의 사상 위에서 세워진 모래성이었습니다:

```
μ :: m ⊗ m -> m
η :: i -> m
```

The reversal of these morphisms produces a comonoid in a monoidal category:

이 초라한 사상들을 가차 없이 뒤집어 버리면 마침내 모노이드 범주 속의 코모노이드란 놈이 괴성을 지르며 튀어나옵니다:

```
δ :: m -> m ⊗ m
ε :: m -> i
```

One can write a definition of a comonoid in Haskell:

혹자는 이 코모노이드 따위를 Haskell로 이렇게 끄적여댈지도 모릅니다:

```
class Comonoid m where
  split   :: m -> (m, m)
  destroy :: m -> ()
```

but it is rather trivial. Obviously destroy ignores its argument.

허나 이건 그저 시시하기 짝이 없을(trivial) 지경입니다. 왜냐면 안 봐도 저 `destroy`라는 망나니는 자기 인수를 개무시해버릴 게 자명하거든요.

```
destroy _ = ()
```

split is just a pair of functions:

`split` 역시도 까고 보면 그저 한 쌍의 하찮은 함수 쪼가리일 뿐입니다:

```
split x = (f x, g x)
```

Now consider comonoid laws that are dual to the monoid unit laws.

이제 그 알량한 모노이드 단위 법칙의 맞은편 거울상인 코모노이드 법칙들을 노려봅시다.

```
lambda . bimap destroy id . split = id
rho . bimap id destroy . split = id
```

Here, lambda and rho are the left and right unitors, respectively (see the definition of monoidal categories). Plugging in the definitions, we get:

여기서 저 `lambda`와 `rho`는 대충 왼쪽 유니터(left unitor)와 오른쪽 유니터(right unitor) 따위를 의미합니다 (이건 저번 모노이드 범주 때의 구질구질한 정의를 참조하십시오). 이 조잡한 정의들을 수식 속에 꾸욱 찔러 넣어 박아보면, 이따위 결과를 얻게 됩니다:

```
lambda (bimap destroy id (split x))
= lambda (bimap destroy id (f x, g x))
= lambda ((), g x)
= g x
```

which proves that g = id. Similarly, the second law expands to f = id. In conclusion:

이로서 `g = id`라는 허탈한 결론이 참으로 어이없게도 증명 되었습니다. 당연하게도 두 번째 법칙의 전개 역시 `f = id`라는 뻔한 소리로 수렴하게 되겠죠. 고로 결론짓자면:

```
split x = (x, x)
```

which shows that in Haskell (and, in general, in the category Set) every object is a trivial comonoid.

이 허무한 결론이 시사하는 바는 바로 Haskell 구역(그리고, 좀 더 보편적인 집합의 범주 `Set` 구역) 안에서 굴러먹는 온갖 잡동사니 대상들은 예외 없이 죄다 시시한(trivial) 코모노이드 따위로 취급된다는 사실입니다.

Fortunately there are other more interesting monoidal categories in which to define comonoids. One of them is the category of endofunctors. And it turns out that, just like the monad is a monoid in the category of endofunctors,

그나마 참으로 하늘이 도운 다행인 점은, 이 시시한 코모노이드 따위를 훨씬 그럴싸하고 흥미진진하게 정의해 낼 만한 여타 색다른 모노이드 범주들이 널려 있다는 사실입니다. 그중 하나가 바로 악명 높은 자기 함자들의 범주(category of endofunctors)입니다. 그리고 여기서도 어김없이, 모나드가 단지 이 자기 함자 범주 끝자락에 매달린 모노이드 찌꺼기였단 진실처럼:

The comonad is a comonoid in the category of endofunctors.

코모나드 역시 그저 자기 함자 범주 구석에 처박혀 징그러운 코모노이드 노릇을 할 뿐이었다는 경악스러운 전말이 드러납니다.

## The Store Comonad

## 코스테이트 코모나드: 스토어 코모나드(The Store Comonad)

Another important example of a comonad is the dual of the state monad. It’s called the costate comonad or, alternatively, the store comonad.

이번에 마주할 아주 묵직한 코모나드의 예시는 바로 상태 모나드(state monad)의 역겨운 쌍대(dual)입니다. 이걸 일컬어 코스테이트 코모나드(costate comonad)라 부르거나, 아니면 덜 유치하게 스토어 코모나드(store comonad)라고도 부릅니다.

We’ve seen before that the state monad is generated by the adjunction that defines the exponentials:

우린 앞서 어떠한 지수(exponentials) 객체를 규정지어주던 수반(adjunction)의 작태가 기어이 상태 모나드를 낳고야 말았던 끔찍한 막장 드라마를 직관한 바 있습니다:

```
L z = z × s
R a = s ⇒ a
```

We’ll use the same adjunction to define the costate comonad. A comonad is defined by the composition L ∘ R:

우린 그 똑같은 수반을 다시 한번 우려먹어 이 코스테이트 코모나드 녀석을 세상에 내놓을 겁니다. 이번엔 정반대로 코모나드가 합체 구조 `L ∘ R`에 의해 정의되겠죠:

```
L (R a) = (s ⇒ a) × s
```

Translating this to Haskell, we start with the adjunction between the Product functor on the left and the Reader functor or the right. Composing Product after Reader is equivalent to the following definition:

이걸 Haskell의 말장난으로 억지로 치환해 보면, 왼쪽의 곱(Product) 함자와 오른쪽의 리더(Reader) 함자 놈들이 벌이는 수반 놀음에서 판이 다시 굴러갑니다. 리더 함자가 훑고 지나간 자리를 곱 함자가 나중에 쓱싹 쓸어 담는 이 합성은 결국 다음의 뻔한 정의와 완벽하게 동치입니다:

```
data Store s a = Store (s -> a) s
```

The counit of the adjunction taken at the object a is the morphism:

어떤 가련한 대상 `a` 하나가 인질로 잡혔을 때, 그 수반 구조가 내놓는 여단일원(counit) 사상의 꼴은 이렇습니다:

```
εa :: ((s ⇒ a) × s) -> a
```

or, in Haskell notation:

혹은 Haskell의 역겨운 표기법으론 이렇게 쓰입니다:

```
counit (Prod (Reader f) s) = f s
```

This becomes our extract:

이 녀석이 고스란히 우리의 추출(`extract`) 함수가 되어줍니다:

```
extract (Store f s) = f s
```

The unit of the adjunction is:

그리고 그 수반의 단일원(unit)은 또 이러합니다:

```
unit :: a -> Reader s (Product a s)
unit a = Reader (\s -> Prod a s)
```

We construct δ, or duplicate, as the horizontal composition:

이제 저 거창한 수평 합성을 들이대어 `δ`, 즉 복제(`duplicate`) 녀석을 조립해 봅시다:

```
δ :: L ∘ R -> L ∘ R ∘ L ∘ R
δ = L ∘ η ∘ R
```

We have to sneak η through the leftmost L, which is the Product functor. It means acting with η, or Store f, on the left component of the pair (that’s what fmap for Product would do). We get:

우리는 지금 저 `η`를 맨 좌측 끄트머리에 버티고 선 `L` — 즉 저 멍청한 곱(Product) 함자 — 너머로 어떻게든 스리슬쩍 밀어 넣어 잠입시켜야만 합니다. 그 짓거리의 본질이란 쌍(pair)의 좌측 성분 위에다 대고 냅다 저 `η` 혹여는 `Store f` 나부랭이를 작용시킨다는 뜻입니다 (기실 저게 곱 함자용으로 벼려진 `fmap`이 할 짓거리겠죠). 그리하여 손에 쥐게 되는 건 이렇습니다:

```
duplicate (Store f s) = Store (Store f) s
```

(Remember that, in the formula for δ, L and R stand for identity natural transformations whose components are identity morphisms.)

(명심하십시오, 아까 저 `δ`의 공식에 등장했던 `L`과 `R` 찌끄러기들은 어디까지나 그 알량한 성분들이 항등 사상이랍시고 행세하는 속 빈 강정 같은 항등 자연 변환(identity natural transformations)들을 대신해 세워둔 허수아비일 뿐입니다.)

Here’s the complete definition of the Store comonad:

이제 여지껏 긁어모은 부스러기들을 기워 만든 그 스토어 코모나드(Store comonad)의 온전한 정의 대령입니다:

```
instance Comonad (Store s) where
  extract (Store f s) = f s
  duplicate (Store f s) = Store (Store f) s
```

You may think of the Reader part of Store as a generalized container of as that are keyed using elements of the type s. For instance, if s is Int, Reader Int a is an infinite bidirectional stream of as. Store pairs this container with a value of the key type. For instance, Reader Int a is paired with an Int. In this case, extract uses this integer to index into the infinite stream. You may think of the second component of Store as the current position.

여러분은 어쩌면 이 스토어(Store)의 일부분인 저 리더(Reader) 따위를, 고작 `s`라는 타입의 쪼가리들을 열쇠(키값) 삼아 치장해 둔 아주 대단히 일반화된 `a`들의 용기(container) 중 하나인 양 치부할지도 모르겠습니다. 가령 예를 들어 `s`가 그깟 `Int` 나부랭이라면, `Reader Int a`는 끝도 없이 양방향으로 쏟아져 내리는 역겨운 무한의 `a` 스트림이나 다름없습니다. 헌데 우리의 스토어(Store)는 이 비루한 용기마저도 그 열쇠(`s`) 타입의 어떤 값 한 개와 기어코 억지로 한 쌍으로 엮어(pairs) 버립니다. 예컨대 앞선 `Reader Int a` 따위가 또 하나의 쓸모없는 `Int` 값과 강제로 짝을 맺게 되는 꼴이죠. 이딴 막장 상황에서 추출(`extract`)이 하는 짓거리라곤, 오직 이 덤으로 주어진 정수 쪼가리를 색인(index) 삼아 저 끔찍한 무한 스트림 속으로 파고들어가 단물 하나를 쪽 빨아먹는 게 전부입니다. 기실 여러분은 저 스토어(Store)의 두 번째 구성 요소를, 지금 내가 처박혀있는 '현재 위치(current position)' 척도로 여겨도 결코 틀린 말이 아닐 겁니다.

Continuing with this example, duplicate creates a new infinite stream indexed by an Int. This stream contains streams as its elements. In particular, at the current position, it contains the original stream. But if you use some other Int (positive or negative) as the key, you’d obtain a shifted stream positioned at that new index.

이 지긋지긋한 예시의 연장선상에서 보자면, 복제(`duplicate`)는 또다시 그 망할 `Int` 쪼가리를 색인 삼는 새파란 무한 스트림 하나를 기어이 창조해 내고야 맙니다. 헌데 이 새로운 스트림에는 또 자기 안에 다른 수많은 스트림들이 집어삼켜져 알맹이 노릇을 하고 자빠졌습니다. 특히 소름 돋게도 이 '현재 위치'에 해당하는 좌표를 뒤져보면 거기엔 처음의 원판 스트림이 징그럽게 도사리고 있죠. 하지만 만약 당신이 열쇠랍시고 (음수든 양수든 간에) 뭔가 또 다른 `Int` 값으로 쑤시고 색인해 들어간다면, 그 새로운 위치만큼 한 칸 쩍 밀려나 어긋난(shifted) 또 다른 스트림이 덜컥 낚여 올라올 따름입니다.

In general, you can convince yourself that when extract acts on the duplicated Store it produces the original Store (in fact, the identity law for the comonad states that extract . duplicate = id).

더 얼버무려 큰 틀에서 얘기해 보자면, 복제되어 잔뜩 불어난 그 기괴한 `Store` 위를 추출(`extract`)이 한번 쓱 훑고 지나가면 결국 언제나 또다시 처음의 무고했던 그 오리지널 `Store`가 우스꽝스럽게 툭 튀어나온다는 자명한 진실을 납득하시게 될 겁니다 (사실, 애당초 코모나드의 그 깐깐한 항등 법칙(identity law) 자체가 뻔질나게 `extract . duplicate = id`라며 못 박아두고 있었으니 말입니다).

The Store comonad plays an important role as the theoretical basis for the Lens library. Conceptually, the Store s a comonad encapsulates the idea of “focusing” (like a lens) on a particular substructure of the date type a using the type s as an index. In particular, a function of the type:

스토어 코모나드(Store comonad)는 렌즈(Lens) 라이브러리를 바쳐 올리는 기둥 같은 든든한 이론적 근거지로서 실로 위대한 역할을 맡고 있습니다. 관념적으로 지껄여 보자면, 저 `Store s a` 코모나드라는 녀석은 `a`라는 데이터 타입의 어떤 구석진 하위 구조(substructure) 하나를 콕 집기 위해 저 `s`라는 타입을 한낱 색인용 장작으로 태워먹으며 그곳에 오만상을 몰아넣고 "초점(focus)"을 맞추는 — 마치 볼록 렌즈가 햇빛을 모아 불을 붙이듯 음침한 — 그 지독한 발상을 캡슐처럼 감싸안은 덩어리입니다. 구체적으로 파고들자면, 딱 봐도 이런 타입으로 칠갑을 한 함수가 그 짓을 합니다:

```
a -> Store s a
```

is equivalent to a pair of functions:

이건 완전히 다음과 같은 불행한 함수 한 쌍의 무리와 완벽하게 동치입니다:

```
set :: a -> s -> a
get :: a -> s
```

If a is a product type, set could be implemented as setting the field of type s inside of a while returning the modified version of a. Similarly, get could be implemented to read the value of the s field from a. We’ll explore these ideas more in the next section.

만약 저 `a`가 곱 타입(product type)에 기반한 찌꺼기라면, `set`이란 함수는 필경 그 `a`의 내부를 파고들어 `s` 타입에 해당하는 어떤 불쌍한 필드(field)의 처지를 새롭게 조작(setting)해 버린 뒤 그 만신창이가 된 `a`의 최신 버전을 그대로 내던지는 짓거리로 너끈하게 구현될 수 있습니다. 아주 판에 박힌 듯이 `get` 역시 그 구질구질한 `a` 속에서 `s` 필드의 값을 날름 파먹어 내뱉도록(read) 구현되겠죠. 뭐, 우린 이 빌어먹게 피곤한 구상들을 이다음 챕터에서 아주 골수까지 파먹으며 뒤적거리게 될 예정입니다.
