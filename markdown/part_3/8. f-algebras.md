## F-Algebras

Posted by Bartosz Milewski under ,

This is part 24 of Categories for Programmers. Previously: Comonads. See the Table of Contents.

이것은 프로그래머를 위한 범주론의 24번째 파트입니다. 이전 글: 코모나드(Comonads). 목차 보기.

We’ve seen several formulations of a monoid: as a set, as a single-object category, as an object in a monoidal category. How much more juice can we squeeze out of this simple concept?

우린 여태 모노이드(monoid)란 놈을 두고 참 별의별 다채로운 공식화 작태를 구경해 왔습니다: 한낱 집합 쪼가리로서, 때론 단일 대상 범주(single-object category)로서, 그리고 모노이드 범주 속의 어엿한 덩어리(대상)로서 말입니다. 이쯤 되면 대체 이 알량하고 단순해 빠진 개념 하나를 쥐어짜서 얼마나 더 많은 즙을 우려낼 수 있을지 가늠조차 안 갈 지경입니다.

Let’s try. Take this definition of a monoid as a set m with a pair of functions:

어디 한번 쥐어짜 봅시다. 자, 모노이드를 그저 한 쌍의 하찮은 함수들을 곁들인 알량한 집합 `m` 따위로 간주해버리는 이 원초적인 정의부터 늘어놓고 시작해 보죠:

```
μ :: m × m -> m
η :: 1 -> m
```

Here, 1 is the terminal object in Set — the singleton set. The first function defines multiplication (it takes a pair of elements and returns their product), the second selects the unit element from m. Not every choice of two functions with these signatures results in a monoid. For that we need to impose additional conditions: associativity and unit laws. But let’s forget about that for a moment and just consider “potential monoids.” A pair of functions is an element of a cartesian product of two sets of functions. We know that these sets may be represented as exponential objects:

여기서 `1`이란 건 집합 범주(`Set`)에 똬리를 튼 끝 대상(terminal object) — 바로 그 놈의 싱글톤(singleton) 집합입니다. 첫 번째 함수는 당연하게도 곱셈(multiplication)을 정의해 냅니다 (건방지게 한 쌍의 알맹이들을 채가서 지지고 볶은 뒤 그 곱의 결과물을 토해내죠). 두 번째 함수는 그저 `m` 속에서 비굴하게 단일원(unit element)을 콕 집어 골라내는 역할일 뿐입니다. 물론 이딴 허접한 서명을 띤 두 개의 함수 쪼가리를 아무렇게나 주워다 짝지어 놓는다고 다 모노이드가 되는 기적이 일어나진 않습니다. 진정한 모노이드로 승격시키려면 그 위에다 결합 법칙과 단위 법칙이라는 더 끔찍하고 속박적인 굴레(추가 조건)들을 강제(impose)해야만 하죠. 하지만 딱 지금 이 순간만큼은 그런 골치 아픈 제약 따윈 뇌 속에서 깨끗이 지워버리고 막연히 "잠재적인 모노이드(potential monoids)"에 대해서만 씨부려봅시다. 한 쌍의 함수란 건 결국 두 함수 집합 간의 빌어먹을 데카르트 곱(cartesian product)이 빚어낸 한낱 원소 쪼가리에 불과합니다. 우린 이 짓궂은 집합들이 다음과 같이 망할 지수 객체(exponential objects)의 몰골로 둔갑해 표현될 수 있다는 걸 이미 뼈저리게 알고 있습니다:

```
μ ∈ m m×m
η ∈ m1
```

The cartesian product of these two sets is:

이 저주받을 두 집합의 데카르트 곱은 이렇습니다:

```
m m×m × m1
```

Using some high-school algebra (which works in every cartesian closed category), we can rewrite it as:

고등학교 수준의 역겨운 대수학(놀랍게도 이건 모든 데카르트 닫힌 범주(cartesian closed category)에서 기가 막히게 통용됩니다)을 좀 지저분하게 비벼보면, 이걸 다음과 같이 스리슬쩍 재작성해 볼 수 있습니다:

```
m m×m + 1
```

The plus sign stands for the coproduct in Set. We have just replaced a pair of functions with a single function — an element of the set:

저놈의 더하기(+) 기호는 집합 범주(`Set`) 구석의 쌍대곱(coproduct)을 뻔뻔스레 상징합니다. 방금 우리는 순식간에 두 개의 알량한 함수 쌍을 통째로 뭉둥그려 달랑 하나의 함수 — 즉 바로 이 집합의 원소 하나 — 로 치환해버리는 범죄를 저지른 겁니다:

```
m × m + 1 -> m
```

Any element of this set of functions is a potential monoid.

이 기형적인 함수 집합의 그 어떤 원소든 간에, 다 잠재적인 모노이드(potential monoid) 나부랭이로 취급될 자격이 있습니다.

The beauty of this formulation is that it leads to interesting generalizations. For instance, how would we describe a group using this language? A group is a monoid with one additional function that assigns the inverse to every element. The latter is a function of the type m->m. As an example, integers form a group with addition as a binary operation, zero as the unit, and negation as the inverse. To define a group we would start with a triple of functions:

이딴 변태적인 공식화 작태가 뿜어내는 기괴한 아름다움은, 이게 우리를 사뭇 쏠쏠하고 흥미로운 일반화(generalizations)의 수렁으로 인도해 준다는 데에 있습니다. 가령 예를 들어, 우린 이 비루한 언어 쪼가리를 가지고 군(group)이란 걸 어떻게 주워섬길 수 있을까요? 군이란 건 그저 모노이드에다, 모든 잡동사니 알맹이마다 일일이 역원(inverse)을 갖다 바치는 함수 하나를 덤으로 얹어놓은 잡탕일 뿐입니다. 이 후자의 함수는 뻔하게도 `m->m` 따위의 타입을 가진 허접한 함수죠. 예를 들어, 정수(integers) 나부랭이들은 덧셈을 이항 연산 삼아, 0을 단일원 삼아, 그리고 부호 반전(negation)을 역원 삼아 구질구질한 하나의 군을 형성해 냅니다. 고로 군 하나를 기필코 정의해 내려면 우린 필경 다음과 같은 세 마리의 함수 트리오(triple)를 집어 들고 시작해야 할 겁니다:

```
m × m -> m
m -> m
1 -> m
```

As before, we can combine all these triples into one set of functions:

아까 써먹었던 야비한 수법을 재탕하여, 우린 이 세 마리의 짐승들을 또다시 단일한 함수 집합이라는 큰 냄비 하나에 몽땅 들이붓고 합칠 수 있습니다:

```
m × m + m + 1 -> m
```

We started with one binary operator (addition), one unary operator (negation), and one nullary operator (identity — here zero). We combined them into one function. All functions with this signature define potential groups.

우린 처음엔 건방진 이항 연산자 무랭이(덧셈), 조금 만만한 단항 연산자(부호 반전), 그리고 속 빈 강정 같은 영항 연산자(nullary operator, 즉 단일원 — 여기선 0)를 나란히 세워두고 삽을 펐습니다. 그러고는 이걸 무자비하게 짓이겨 단 하나의 함수 덩어리로 합쳐냈죠. 바로 이 흉측한 서명을 품은 잡다한 함수들은 여하튼 죄다 잠재적인 군(potential groups)들을 읊어대는 셈입니다.

We can go on like this. For instance, to define a ring, we would add one more binary operator and one nullary operator, and so on. Each time we end up with a function type whose left-hand side is a sum of powers (possibly including the zeroth power — the terminal object), and the right-hand side being the set itself.

우린 이딴 징그러운 막장 짓거리를 끝도 없이 계속해 나갈 수 있습니다. 가령 환(ring)을 하나 빚어내려면, 거기에다 이항 연산자 하나랑 영항 연산자 하나 따위를 무책임하게 툭 던져 추가하는 식이죠. 이 짓거리를 반복할 때마다 우린 결국 구역질 나게도, 그 좌변엔 온갖 거듭제곱들의 지저분한 합(sum of powers; 어쩌면 0제곱, 즉 끝 대상까지 기어들어 간 꼴로)이 진득하게 눌어붙어 있고, 우변엔 그저 집합 그 자신만 덩그러니 남는 끔찍한 함수 타입 하나와 직면하게 됩니다.

Now we can go crazy with generalizations. First of all, we can replace sets with objects and functions with morphisms. We can define n-ary operators as morphisms from n-ary products. It means that we need a category that supports finite products. For nullary operators we require the existence of the terminal object. So we need a cartesian category. In order to combine these operators we need exponentials, so that’s a cartesian closed category. Finally, we need coproducts to complete our algebraic shenanigans.

자, 이제 이참에 아주 작정하고 광기 어린 일반화의 폭주를 달려봅시다. 우선 첫 빠따로 촌스러운 집합 따위를 모조리 걷어치우고 그 자릴 거창한 대상(objects)들로 채우며, 싸구려 함수들은 전부 고결한 사상(morphisms)으로 확 갈아치웁니다. `n`항 연산자 따위는 `n`항 곱에서부터 기어 나오는 사상으로 둔갑시켜 정의해 버리면 그만입니다. 이 미친 수작이 뜻하는 바는 즉, 우리가 유한한 곱(finite products) 나부랭이를 뒷바라지해 줄 어느 호구 같은 범주를 수배해야만 한단 소리입니다. 거기에 영항 연산자까지 써먹자니 좆같이도 끝 대상(terminal object)이 실존해야만 한답니다. 그러니 우린 데카르트 범주(cartesian category)가 몹시도 궁해집니다. 이 제멋대로 튀는 연산자 놈들을 한데 쓸어 담고 엮어내려면 이번엔 지수(exponentials)까지 등판해야 하니, 이 바닥이 갈수록 꼬여 결국 그건 데카르트 닫힌 범주(cartesian closed category)가 되고야 마네요. 대망의 피날레로 우리의 이 지저분한 대수적 난동(algebraic shenanigans)을 찬란하게 완성 짓기 위해선 기어코 쌍대곱(coproducts)까지 제물로 바쳐야 합니다.

Alternatively, we can just forget about the way we derived our formulas and concentrate on the final product. The sum of products on the left hand side of our morphism defines an endofunctor. What if we pick an arbitrary endofunctor F instead? In that case we don’t have to impose any constraints on our category. What we obtain is called an F-algebra.

혹은, 우리 손모가지로 그 역겨운 공식들을 주물러 도출해 냈던 그 더러운 과거의 구질구질한 궤적일랑은 개나 줘버리고 뇌에서 싹 지운 채, 오로지 이 최종 산물(결과물)에만 침을 흘리며 노려볼 수도 있겠습니다. 저 사상의 좌변을 그득 채운 알량한 곱들의 합(sum of products)이란 꼬락서니가 기실 하나의 자기 함자(endofunctor)를 버젓이 정의해 둡니다. 까짓거 그따위 것들 말고 냅다 무작위의 아무렇게나 굴러먹는 개잡 함자 `F`를 속 편하게 하나 집어 들면 어떻게 될까요? 만약 그런 막장 사태라면, 우린 더 이상 우리의 가엾은 범주 위로 저딴 더러운 구속과 제약(constraints) 따위를 일일이 강요할 필요조차 완전히 증발해 버립니다. 이 광란의 질주 끝에 우리 수중에 뚝 떨어지는 기괴한 전리품, 그걸 우린 F-대수(F-algebra)라 부릅니다.

An F-algebra is a triple consisting of an endofunctor F, an object a, and a morphism

하나의 F-대수란 끝끝내 자기 함자 `F` 나부랭이 하나와 어떤 대상 `a`, 그리고 하나의 불운한 사상으로 뒤엉켜 직조된 3인조 트리오(triple)일 뿐입니다:

```
F a -> a
```

The object is often called the carrier, an underlying object or, in the context of programming, the carrier type. The morphism is often called the evaluation function or the structure map. Think of the functor F as forming expressions and the morphism as evaluating them.

여기 속한 대상은 흔히 운반자(carrier)니, 기저 대상(underlying object) 따위로 불리거나 프로그래밍의 척박한 문맥 속에서는 가볍게 캐리어 타입(carrier type)이라 지껄여집니다. 저 사상은 흔히 평가 함수(evaluation function) 혹은 구조 사상(structure map)이라 포장되곤 하죠. 저 함자 `F`라는 녀석은 고작 표식(수식, expressions)들 따위를 빚어내는 잡일이나 하고, 반면에 저 사상은 그걸 무자비하게 평가절하하여 실체를 산출해 낸다(evaluating)고 대충 퉁쳐 생각하시면 됩니다.

Here’s the Haskell definition of an F-algebra:

여기 Haskell 판에 굴러다니는 그 잘난 F-대수의 면상(정의) 대령입니다:

```
type Algebra f a = f a -> a
```

It identifies the algebra with its evaluation function.

이건 기어코 그 대수(algebra) 자체를 평가 함수라는 껍데기 하나와 완전히 동일선상에 처박고 동일시(identifies) 해 버립니다.

In the monoid example, the functor in question is:

앞서 들먹였던 그 모노이드 예제에서, 우리가 끄집어냈던 문제의 함자란 게 실체는 이러했습니다:

```
data MonF a = MEmpty | MAppend a a
```

This is Haskell for 1 + a × a (remember algebraic data structures).

이거야말로 그 알량한 대수적 데이터 구조(algebraic data structures)들을 들먹이던 시절의 그 `1 + a × a` 수식을 Haskell의 시궁창 언어로 토해낸 겁니다(기억부터 좀 되짚어보세요).

A ring would be defined using the following functor:

환(ring) 나부랭이를 정의하려면 필경 이딴 역겨운 함자를 갖다 써야만 할 겁니다:

```
data RingF a = RZero
             | ROne
             | RAdd a a 
             | RMul a a
             | RNeg a
```

which is Haskell for 1 + 1 + a × a + a × a + a.

이 미친 괴물은 다름 아닌 저 `1 + 1 + a × a + a × a + a`를 Haskell 방언으로 게워낸 쓰레기 덩어리입니다.

An example of a ring is the set of integers. We can choose Integer as the carrier type and define the evaluation function as:

환의 그나마 쓸모 있는 예시 하나는 다름 아닌 정수 집합(set of integers)입니다. 우린 캐리어(운반자) 타입이랍시고 오만하게 `Integer`를 지명한 다음, 그 멍청한 평가 함수를 이따위로 끄적여댈 수 있습니다:

```
evalZ :: Algebra RingF Integer
evalZ RZero      = 0
evalZ ROne       = 1
evalZ (RAdd m n) = m + n
evalZ (RMul m n) = m * n
evalZ (RNeg n)   = -n
```

There are more F-algebras based on the same functor RingF. For instance, polynomials form a ring and so do square matrices.

저 잘난 동일한 `RingF` 함자를 기반으로 빨대 꽂고 기생하는 다른 수많은 F-대수들이 이 세상 천지에 널려 있습니다. 가령 다항식(polynomials)들도 시건방지게 독자적인 하나의 환을 쳐 형성하고 있고, 빌어먹을 정사각 행렬(square matrices) 무리들 역시 똑같은 작태를 벌이고 자빠졌죠.

As you can see, the role of the functor is to generate expressions that can be evaluated using the evaluator of the algebra. So far we’ve only seen very simple expressions. We are often interested in more elaborate expressions that can be defined using recursion.

두 눈알로 똑똑히 목격하다시피, 함자가 이 짓거리에서 맡은 비루한 임무라곤 고작해야 그 콧대 높은 대수의 평가자(evaluator) 입에 떠먹여 줄 시시콜콜한 수식(expressions) 쪼가리들을 열심히 생성해 내는 것뿐입니다. 여지껏 우린 그저 몹시도 수준 낮고 단순무식한 수식들만 구경해 왔습니다. 허나 으레 그렇듯 우리의 빌어먹을 흥미는 곧 재귀(recursion) 따위를 마구 뒤섞어 정의해야만 속이 풀릴 법한, 한층 더 정교하고 악랄한 수식들을 향해 뒤틀리기 마련입니다.

## Recursion

## 재귀(Recursion)

One way to generate arbitrary expression trees is to replace the variable a inside the functor definition with recursion. For instance, an arbitrary expression in a ring is generated by this tree-like data structure:

변덕스러운 임의의 수식 트리(expression trees)들을 무작위로 찍어내는 방편 중 하나는, 저 함자의 정의 속에 틀어박힌 변수 `a` 쪼가리를 아예 돌았다고 역겨운 재귀(recursion) 덩어리로 통째로 갈아치워 버리는 겁니다. 가령 예컨대, 어떤 환 속의 임의의 수식 나부랭이는 아래와 같은 나무뿌리 닮은(tree-like) 데이터 구조로 지저분하게 연성됩니다:

```
data Expr = RZero
          | ROne
          | RAdd Expr Expr 
          | RMul Expr Expr
          | RNeg Expr
```

We can replace the original ring evaluator with its recursive version:

우린 앞서 써먹던 케케묵은 환(ring)의 평가자 녀석을, 이딴 끔찍한 재귀 버전으로 냉큼 치워 바꿀 수 있습니다:

```
evalZ :: Expr -> Integer
evalZ RZero        = 0
evalZ ROne         = 1
evalZ (RAdd e1 e2) = evalZ e1 + evalZ e2
evalZ (RMul e1 e2) = evalZ e1 * evalZ e2
evalZ (RNeg e)     = -(evalZ e)
```

This is still not very practical, since we are forced to represent all integers as sums of ones, but it will do in a pinch.

이건 까놓고 말해 여전히 쓸모가 쥐뿔도 없는 짓거리지만 (시발 툭하면 온갖 정수들을 죄다 '1'들의 합산 쪼가리들로만 토해내도록 구속받기 때문이죠), 그래도 아쉬운 대로 급한 불을 끄는 덴(in a pinch) 그럭저럭 써먹을 만합니다.

But how can we describe expression trees using the language of F-algebras? We have to somehow formalize the process of replacing the free type variable in the definition of our functor, recursively, with the result of the replacement. Imagine doing this in steps. First, define a depth-one tree as:

허나 우리는 대체 저 수식 트리들이란 놈들을 무슨 수로 망할 F-대수의 언어를 동원해 묘사해 낼 수 있느냐는 겁니다? 여하튼 어떻게든 우리는 함자의 정의 속에 숨어 있는 자유 타입 변수(free type variable) 놈을 파내고, 그 자리에 재귀적인 치환의 결과물(result of the replacement)을 쳐박는 그 역겨운 행각을 보란 듯이 형식화(formalize)해 놔야만 합니다. 이 짓거리를 단계별로 끊어서 상상해 보십시오. 우선 깊이-1 찌꺼기 트리를 이따위로 정의하는 겁니다:

```
type RingF1 a = RingF (RingF a)
```

We are filling the holes in the definition of RingF with depth-zero trees generated by RingF a. Depth-2 trees are similarly obtained as:

우리는 지금 저 `RingF a` 따위가 싸질러놓은 깊이-0 따리의 파편들로 `RingF`의 뚫린 빵꾸 구멍들을 성실히 처박아 메우는 중입니다. 그와 똑같은 작태로 깊이-2 트리는 또 이렇게 토혈해 냅니다:

```
type RingF2 a = RingF (RingF (RingF a))
```

which we can also write as:

이걸 또 이렇게 재활용해서 갈겨 쓸 수도 있습니다:

```
type RingF2 a = RingF (RingF1 a)
```

Continuing this process, we can write a symbolic equation:

이 노가다를 지긋지긋하게 반복하다 보면, 우린 기어이 다음과 같은 기호 방정식(symbolic equation) 하나에 도달해 끄적거리게 됩니다:

```
type RingFn+1 a = RingF (RingFn a)
```

Conceptually, after repeating this process infinitely many times, we end up with our Expr. Notice that Expr does not depend on a. The starting point of our journey doesn’t matter, we always end up in the same place. This is not always true for an arbitrary endofunctor in an arbitrary category, but in the category Set things are nice.

관념적으로 따지자면, 이 악마 같은 짓거리를 무한대의 횟수로 반복하고 나면 우린 끝내 우리의 종착역인 저 `Expr`에 도달하고 맙니다. 헌데 이 `Expr`이란 작자는 가만 보면 저 `a` 나부랭이 따위엔 눈곱만큼도 의존하지 않는다는 걸 눈여겨보십시오. 우리의 고난이 뻗어나간 그 첫 출발점 따위는 도무지 알 바가 아니고, 우린 이내 항상 똑같은 구덩이 속으로 처박힌다는 소리입니다. 물론 무작위로 굴러먹는 임의의 범주 속 임의의 자기 함자들에게 이 말이 무조건 진리랍시고 통하는 건 아니지만, 그나마 저 얌전한 `Set` 범주 속에서만큼은 상황이 무척이나 달달하게(nice) 돌아갑니다.

Of course, this is a hand-waving argument, and I’ll make it more rigorous later.

뭐 당연하게도 이딴 논리 전개는 그냥 대충 손이나 휙휙 저으면서(hand-waving) 사기 치는 수준의 날림 주장이고, 좀 더 피 흘리는 엄밀한 증명은 나중에 때가 되면 마저 때려 넣겠습니다.

Applying an endofunctor infinitely many times produces a fixed point, an object defined as:

여하튼 자기 함자를 향해 그 무자비한 짓거리를 무한히 적용하고 나면, 끝내 하나의 고정점(fixed point) — 다음과 같이 기괴하게 정의되는 대상 — 하나를 떡하니 산출해 냅니다:

```
Fix f = f (Fix f)
```

The intuition behind this definition is that, since we applied f infinitely many times to get Fix f, applying it one more time doesn’t change anything. In Haskell, the definition of a fixed point is:

이 저주받을 정의 뒤통수에 도사리고 있는 직관이란 뻔합니다. 우리가 저놈의 `Fix f`인지 뭔지를 얻겠답시고 이미 그깟 `f`를 무식하게 무한히 갖다 박았는데, 거기다 대고 기껏 한 번쯤 더 바른다고 한들 시발 세상천지에 변할 게 뭐가 있겠냐는 겁니다. Haskell의 언어로 이 고정점의 실체를 토해내면 이렇습니다:

```
newtype Fix f = Fix (f (Fix f))
```

Arguably, this would be more readable if the constructor’s name were different than the name of the type being defined, as in:

까고 말해, 행여 저 빌어먹을 생성자(constructor) 이름 따위를 지금 정의되는 타입의 면상과 아예 별개로 지어버렸으면 그나마 좀 알아먹기 나았을지도 모를 일입니다, 이런 식으로 말이죠:

```
newtype Fix f = In (f (Fix f))
```

but I’ll stick with the accepted notation. The constructor Fix (or In, if you prefer) can be seen as a function:

허나 난 그냥 세간에 흔히 받아들여지는 썩은 관행(표기법)을 고집스레 핥고 갈 작정입니다. 이 생성자 `Fix`는 (아니면 당신 꼴리는 대로 `In`이라 부르든가) 대충 이런 함수의 꼴로 치부될 수 있습니다:

```
Fix :: f (Fix f) -> Fix f
```

There is also a function that peels off one level of functor application:

여기에 더해, 이 겹겹이 쌓인 함자 표피의 한 꺼풀을 시원하게 벗겨버리는 얄팍한 함수도 하나 존재합니다:

```
unFix :: Fix f -> f (Fix f)
unFix (Fix x) = x
```

The two functions are the inverse of each other. We’ll use these functions later.

당연하게도 이 두 함수 놈들은 지들끼리 완벽한 역(inverse)의 관계에 놓여 있습니다. 이 잡기계들은 나중을 기약하며 짱박아두겠습니다.

## Category of F-Algebras

## F-대수들의 범주(Category of F-Algebras)

Here’s the oldest trick in the book: Whenever you come up with a way of constructing some new objects, see if they form a category. Not surprisingly, algebras over a given endofunctor F form a category. Objects in that category are algebras — pairs consisting of a carrier object a and a morphism F a -> a, both from the original category C.

이 바닥에서 가장 케케묵고 울궈먹기 좋은 낡은 수법(trick) 하나 알려드리죠: 당신이 어떤 기괴한 새 대상(objects)들을 찍어내는 수작을 창안해 낼 때마다, 그 폐급들이 모여서 범주(category)를 형성할 수나 있는지 냉큼 시험해 보는 겁니다. 놀랄 것도 없이, 주어진 어떤 자기 함자 `F` 위에서 둥지를 튼 대수(algebras)들은 버젓이 하나의 범주를 형성해냅니다. 그 빌어먹을 범주 안의 대상들이란 바로 저 대수들이며 — 본래의 범주 `C`에서 끌어온 캐리어 대상 `a` 하나와 그 사상 `F a -> a` 나부랭이가 질척거리며 엉킨 한 쌍(pairs)들에 불과합니다.

To complete the picture, we have to define morphisms in the category of F-algebras. A morphism must map one algebra (a, f) to another algebra (b, g). We’ll define it as a morphism m that maps the carriers — it goes from a to b in the original category. Not any morphism will do: we want it to be compatible with the two evaluators. (We call such a structure-preserving morphism a homomorphism.) Here’s how you define a homomorphism of F-algebras. First, notice that we can lift m to the mapping:

이 헛된 그림을 완성시키려거든, 우린 마저 이 F-대수들의 범주 속에서 나부낄 사상(morphisms) 따위를 억지로라도 정의해 내야만 합니다. 하나의 사상은 응당 어떤 대수 `(a, f)`를 또 다른 대수 `(b, g)`로 무자비하게 매핑(map)시켜야만 합니다. 우린 이걸 캐리어 대상들을 매핑해대는 어떤 사상 `m`인 양 조작할 것입니다 — 즉, 처음의 그 원판 범주 속에서 `a`로부터 `b`로 건너가는 짓거리죠. 물론 조잡한 아무 사상이나 다 이 특혜를 누릴 순 없습니다: 우린 그 녀석이 저 두 평가자(evaluators) 놈들과 서로 아귀가 짝짝 들어맞으며 양립(compatible)하길 원합니다. (이딴식으로 얍삽하게 구조를 보존해 내는 사상 따위를 우린 준동형 사상(homomorphism)이라 지칭합니다.) 자, 여기 F-대수의 준동형 사상을 주조해 내는 레시피가 있습니다. 우선, 우린 저 사상 `m` 놈의 멱살을 거뜬히 들어올려(lift) 이런 매핑으로 탈바꿈시킬 수 있다는 사실을 주지하십시오:

```
F m :: F a -> F b
```

we can then follow it with g to get to b. Equivalently, we can use f to go from F a to a and then follow it with m. We want the two paths to be equal:

그런 다음 그 뒤에다 `g`를 질질 달고선 마침내 저 `b`에 도달할 수 있습니다. 이것과 완전히 동치인 짓거리로, 이번엔 `f`를 징검다리 삼아 `F a`에서 `a`로 먼저 기어 내려온 뒤, 그 꽁무니를 물고 `m`을 쫓아가는 수작도 부릴 수 있죠. 우리는 이 치졸한 두 경로가 기어이 같은 길이어야만(equal) 한다고 억지를 부립니다:

```
g ∘ F m = m ∘ f
```

It’s easy to convince yourself that this is indeed a category (hint: identity morphisms from C work just fine, and a composition of homomorphisms is a homomorphism).

이 허접한 작태가 실은 버젓한 하나의 범주라는 사실엔 아주 식은 죽 먹기로 수긍하게 될 겁니다 (힌트: `C` 놈에게서 강탈해온 항등 사상 찌꺼기들이 아주 멀쩡히 작동하며, 준동형 사상들끼리 지지고 볶고 합성해 봤자 튀어나오는 건 결국 또 다른 준동형 사상 한 마리랍니다).

An initial object in the category of F-algebras, if it exists, is called the initial algebra. Let’s call the carrier of this initial algebra i and its evaluator j :: F i -> i. It turns out that j, the evaluator of the initial algebra, is an isomorphism. This result is known as Lambek’s theorem. The proof relies on the definition of the initial object, which requires that there be a unique homomorphism m from it to any other F-algebra. Since m is a homomorphism, the following diagram must commute:

F-대수의 범주 안에 처박혀있는 초기 대상(initial object) 따위가 만약 실존하기만 한다면, 그걸 우린 초기 대수(initial algebra)라 일컫습니다. 이 잘난 초기 대수의 캐리어를 `i`라 칭하고, 그 평가자는 `j :: F i -> i`라고 갖다 붙입시다. 경악스럽게도 저 초기 대수의 평가자 `j`라는 놈은 결국 동형사상(isomorphism)이라는 전말이 탄로납니다. 이 해괴한 촌극의 결과물을 람벡의 정리(Lambek's theorem)라 부릅니다. 이 자명해 보이는 정리의 증명 따윈 오직 자신으로부터 여타 모든 F-대수들을 향해 유일무이한(unique) 준동형 사상 `m` 하나만을 토해내야 한다는, 초기 대상의 그 깐깐한 정의 나부랭이에 전적으로 기생합니다. `m`이 엄연히 준동형 사상인 이상, 다음에 널부러진 다이어그램은 반드시 교환(commute)해야만 합니다:

Now let’s construct an algebra whose carrier is F i. The evaluator of such an algebra must be a morphism from F (F i) to F i. We can easily construct such an evaluator simply by lifting j:

자, 이제 그 잘난 캐리어가 하필이면 무려 `F i` 씩이나 되는 조악한 대수를 하나 조립해 봅시다. 이딴 정신 나간 대수의 평가자는 필경 `F (F i)`에서 `F i`로 처박히는 사상이어야만 할 겁니다. 우린 단지 저 `j`의 멱살을 확 움켜쥐고 들어올려(lifting) 주는 것만으로도 이딴 허접한 평가자를 눈 감고도 빚어낼 수 있습니다:

```
F j :: F (F i) -> F i
```

Because (i, j) is the initial algebra, there must be a unique homomorphism m from it to (F i, F j). The following diagram must commute:

저 `(i, j)`가 당당히 초기 대수로 군림하는 이상, 거기서부터 저 기형적인 `(F i, F j)`로 뻗어나가는 유일무이한 준동형 사상 `m`이 필연적으로 도사리고 있어야만 합니다. 즉 이 아래의 다이어그램 역시 불만 없이 닥치고 교환되어야 한다는 뜻이죠:

But we also have this trivially commuting diagram (both paths are the same!):

허나 우린 이런 어이없게도 시시해서 하품만 나오는 교환 다이어그램(양 갈래 길이 시발 완벽히 똑같거든요!) 하나를 덤으로 쥐고 있죠:

which can be interpreted as showing that j is a homomorphism of algebras, mapping (F i, F j) to (i, j). We can glue these two diagrams together to get:

이 맹탕 같은 그림 쪼가리를 어떻게 억지로 해석해 보자면, 저 `j`라는 작자가 실은 `(F i, F j)`를 날름 집어삼켜 `(i, j)`로 뱉어놓는 어엿한 대수들의 준동형 사상 노릇을 해내고 있다는 걸 증명하는 셈입니다. 이 쓰레기 같은 두 다이어그램 판자때기를 더덕더덕 이어 붙이면 이딴 걸 얻어맞게 됩니다:

This diagram may, in turn, be interpreted as showing that j ∘ m is a homomorphism of algebras. Only in this case the two algebras are the same. Moreover, because (i, j) is initial, there can only be one homomorphism from it to itself, and that’s the identity morphism idi — which we know is a homomorphism of algebras. Therefore j ∘ m = idi. Using this fact and the commuting property of the left diagram we can prove that m ∘ j = idFi. This shows that m is the inverse of j and therefore j is an isomorphism between F i and i:

이 기괴한 다이어그램 덩어리 역시 순서대로 뜯어보자면, `j ∘ m`이라는 합체 로봇이 기실 대수들의 준동형 사상 짓을 하고 자빠졌다는 폭로로 받아들여집니다. 단 이 골 때리는 상황에선 놀랍게도 출발지와 목적지의 두 대수가 완벽히 똑같은 놈이라는 게 문제입니다. 덤으로 저 위대한 `(i, j)`는 자타가 공인하는 초기(initial) 객체인지라, 자기 자신에게로 꺾여 돌아오는 준동형 사상 따위는 천지간에 달랑 하나뿐이어야 마땅하고, 그게 바로 대수 준동형 사상의 탈을 쓴 영광스러운 항등 사상 `idi`일 수밖에 없습니다. 고로 `j ∘ m = idi`라는 유치한 산수에 다다르죠. 이 시시한 사실과 바로 직전 왼쪽 다이어그램에서 우려먹은 교환 성질(commuting property)을 비벼보면, 우리는 기어코 `m ∘ j = idFi`라는 명제까지 증명해 낼 수 있습니다. 이 역겨운 말장난들의 참뜻은, 저 `m`이 사실은 `j`의 아가리에 처박힌 역(inverse) 찌꺼기이며, 결과적으로 `j`란 놈은 `F i`와 `i` 사이를 미친 듯이 오고 가는 동형사상(isomorphism)이라는 끔찍한 진리입니다:

```
F i ≅ i
```

But that is just saying that i is a fixed point of F. That’s the formal proof behind the original hand-waving argument.

근데 이딴 소린 까놓고 말해 저 `i`가 그냥 `F`의 고정점(fixed point)이라는 사실을 길고 지루하게 게워낸 것에 불과합니다. 이게 바로 아까 손이나 대충 휘저어가며 나불댔던 솜사탕 같은 논리의 이면에 숨겨져 있던 역겨울 만큼 엄밀한 공식 증명(formal proof) 되시겠습니다.

Back to Haskell: We recognize i as our Fix f, j as our constructor Fix, and its inverse as unFix. The isomorphism in Lambek’s theorem tells us that, in order to get the initial algebra, we take the functor f and replace its argument a with Fix f. We also see why the fixed point does not depend on a.

다시 지긋지긋한 Haskell의 늪으로 돌아가 보죠: 우린 저 `i`를 우리가 애지중지하던 `Fix f`로, 저 `j`를 우리의 그 생성자 `Fix`로, 그리고 그 역(inverse) 찌끄레기를 `unFix`라는 이름으로 친숙하게 인식합니다. 람벡의 정리(Lambek's theorem) 속 동형사상이 우리에게 속삭이는 계시란 이런 겁니다: 그 위대한 초기 대수 나부랭이를 어떻게든 얻어내려거든, 저 함자 `f`를 징발하여 그 빌어먹을 인수 `a`를 기어이 `Fix f` 따위로 송두리째 치환해버리라는 뜻입니다. 또한 왜 그 잘난 고정점(fixed point)이 하찮은 `a` 하나에 얽매이지 않는지도 구역질이 날 만큼 뻔하게 꿰뚫어볼 수 있습니다.

### Natural Numbers

### 자연수(Natural Numbers)

Natural numbers can also be defined as an F-algebra. The starting point is the pair of morphisms:

자연수라는 알량한 녀석들도 그깟 F-대수 무랭이 중 하나로 얼렁뚱땅 정의해버릴 수 있습니다. 그 지긋지긋한 여정의 시발점은 응당 한 쌍의 사상 놈들입니다:

```
zero :: 1 -> N
succ :: N -> N
```

The first one picks the zero, and the second one maps all numbers to their successors. As before, we can combine the two into one:

첫 번째 미친놈은 오로지 0만을 집요하게 골라내고, 두 번째 또라이는 세상의 모든 숫자들을 거칠게 낚아채 그들의 다음 숫자(successors)에게로 사정없이 매핑시켜 버립니다. 늘 하던 수작질처럼, 우리는 또다시 저 두 놈을 포대기에 싸서 단 하나의 괴물로 합체시킬 수 있습니다:

```
1 + N -> N
```

The left hand side defines a functor which, in Haskell, can be written like this:

좌변의 흉물스러운 덩어리는 하나의 함자(functor)를 규정짓는데, 이걸 Haskell 특유의 변태적인 문법으로 갈겨 쓰면 이따위로 나옵니다:

```
data NatF a = ZeroF | SuccF a
```

The fixed point of this functor (the initial algebra that it generates) can be encoded in Haskell as:

이딴 함자의 고정점(fixed point) 나부랭이 — 즉 이 쓸모없는 녀석이 쥐어짜 내는 그 대단하신 초기 대수 — 는 Haskell 구역 안에서 이렇게 조잡하게 암호화(encoded)됩니다:

```
data Nat = Zero | Succ Nat
```

A natural number is either zero or a successor of another number. This is known as the Peano representation for natural numbers.

자연수는 그저 고작 0이거나, 아니면 무언가 다른 비루한 숫자의 따까리(다음 숫자, successor) 따위로 치부될 뿐이라는 소리입니다. 이 시시한 사실은 무려 자연수를 향한 페아노 표현법(Peano representation)이라는 잘난 척하는 이름으로 통용됩니다.

## Catamorphisms

## 카타모피즘(Catamorphisms)

Let’s rewrite the initiality condition using Haskell notation. We call the initial algebra Fix f. Its evaluator is the contructor Fix. There is a unique morphism m from the initial algebra to any other algebra over the same functor. Let’s pick an algebra whose carrier is a and the evaluator is alg.

자, 그 잘난 초기성 조건(initiality condition)이란 걸 Haskell의 기호 빌려다가 싹 다 다시 써 발겨 봅시다. 우린 거창한 초기 대수 따위를 `Fix f`라고 부릅니다. 그놈의 평가자란 건 다름 아닌 `Fix` 생성자이고요. 이 초기 대수 구역에서 동일한 함자를 뒤집어쓴 다른 그 어떤 대수를 향해서든 언제나 단 하나뿐인(unique) 사상 `m`이 쏘아져 나아갑니다. 임의로 한번 캐리어가 `a`이고 평가자가 `alg`인 대수 하나를 콕 집어 골라봅시다.

By the way, notice what m is: It’s an evaluator for the fixed point, an evaluator for the whole recursive expression tree. Let’s find a general way of implementing it.

그건 그렇고, 저 `m`이라는 잡것의 본질이 대체 뭔지나 좀 심도 있게 뜯어보십시오: 저놈은 실상 고정점(fixed point)을 잡아먹는 평가자일 뿐만 아니라, 썩어 문드러진 거대한 재귀적 수식 트리 전체를 깡그리 씹어 삼키는 평가자이기도 합니다. 이제 이 괴물을 우겨넣어 구현해 낼 흔하고 보편적인 방편이나 한번 작정하고 파헤쳐 볼까요.

Lambek’s theorem tells us that the constructor Fix is an isomorphism. We called its inverse unFix. We can therefore flip one arrow in this diagram to get:

람벡의 정리(Lambek's theorem)는 우리에게 저 `Fix` 생성자 따위가 기실 동형사상(isomorphism)이라고 나불댔습니다. 우린 그놈의 역(inverse) 찌꺼기에다 `unFix`라는 꼬리표를 달아줬죠. 고로 우린 이 다이어그램에서 화살표 하나를 냅다 반대로 확 꺾어버려서 이딴 꼬락서니를 얻어맞을 수 있습니다:

Let’s write down the commutation condition for this diagram:

이 지저분한 다이어그램을 두고서 교환의 조건(commutation condition)을 가차 없이 적어 갈겨 봅시다:

```
m = alg . fmap m . unFix
```

We can interpret this equation as a recursive definition of m. The recursion is bound to terminate for any finite tree created using the functor f. We can see that by noticing that fmap m operates underneath the top layer of the functor f. In other words, it works on the children of the original tree. The children are always one level shallower than the original tree.

우린 이 뻔한 방정식을 저 `m`의 면상을 까발리는 재귀적 정의(recursive definition)로 속 편히 우겨 해석할 수 있습니다. 이놈의 재귀 작태란 건, 함자 `f`에 의해 빚어진 그 어떤 유한한(finite) 길이를 가진 트리에서건 간에 반드시 기어코 멸망(terminate)하게끔 운명 지어져 있습니다. 그 이유는 바로 저 구역질 나는 `fmap m`이라는 놈이 함자 `f`의 맨 꼭대기 층(top layer) 바로 발밑, 즉 하부에서만 찌질하게 굴러먹는다는 걸 파악해 보면 쉽게 까발려집니다. 까놓고 말해 저놈은 오직 본래 트리의 불쌍한 자식 노드(children)들 위에서만 알량하게 작동한다는 뜻입니다. 그리고 그 자식 새끼들은 당연하게도 원래의 어미 트리보단 무조건 한 단계씩 깊이가 훨씬 더 얕을(shallower) 수밖에 없으니까요.

Here’s what happens when we apply m to a tree constructed using Fix f. The action of unFix peels off the constructor, exposing the top level of the tree. We then apply m to all the children of the top node. This produces results of type a. Finally, we combine those results by applying the non-recursive evaluator alg. The key point is that our evaluator alg is a simple non-recursive function.

우리가 저 `Fix f` 따위로 조립된 트리에다 냅다 저놈의 `m`을 비벼 바르면 이딴 끔찍한 참사가 벌어집니다. 우선 `unFix`의 만행으로 생성자라는 껍데기가 훌러덩 벗겨져 나가고, 끝내 트리의 맨 꼭대기 층이 벌거벗은 채 노출(exposing)됩니다. 그런 다음 우린 오만하게도 최상단 노드의 모든 자식 새끼들 머리 위로 또다시 저 `m`을 무자비하게 내리꽂습니다. 이 짓의 결과로 필연적으로 `a` 타입의 찌꺼기 산출물들이 바닥에 우수수 떨어집니다. 대망의 피날레로, 우리는 감히 비-재귀적인(non-recursive) 순진무구한 평가자 `alg`를 들이밀어 그 끔찍한 잔해물들을 고스란히 하나로 엮어(combine)냅니다. 여기서 가장 핵심적인 뽀인트는, 우리의 그 잘난 평가자 `alg`라는 놈 그 자체는 아주 단순무식하기 짝이 없는 멍청한 비-재귀적인 함수 나부랭이에 불과하단 사실입니다.

Since we can do this for any algebra alg, it makes sense to define a higher order function that takes the algebra as a parameter and gives us the function we called m. This higher order function is called a catamorphism:

이딴 짓거리를 세상 천지 아무 대수 `alg`한테나 다 똑같이 써먹을 수 있는 마당에, 대수를 그깟 매개변수 삼아 꿀꺽 집어삼키고선 기어이 우리가 부르짖던 저 고결한 사상 `m`을 토해내 주는 고차 함수(higher order function)를 대놓고 정의해 버리는 것도 심히 사리에 맞을 겁니다. 이 거창한 고차 함수야말로 이 바닥의 진정한 카타모피즘(catamorphism)이라 칭송받습니다:

```
cata :: Functor f => (f a -> a) -> Fix f -> a
cata alg = alg . fmap (cata alg) . unFix
```

Let’s see an example of that. Take the functor that defines natural numbers:

어디 그 빌어먹을 예시나 하나 맛봅시다. 아까 자연수를 정의해 댔던 그놈의 함자를 다시 끌어와 보죠:

```
data NatF a = ZeroF | SuccF a
```

Let’s pick (Int, Int) as the carrier type and define our algebra as:

이번엔 아예 캐리어 타입이랍시고 만만하게 `(Int, Int)`를 콕 한 번 집어 들고, 우리의 알량한 대수를 이따위 꼴로 정의해 보겠습니다:

```
fib :: NatF (Int, Int) -> (Int, Int)
fib ZeroF = (1, 1)
fib (SuccF (m, n)) = (n, m + n)
```

You can easily convince yourself that the catamorphism for this algebra, cata fib, calculates Fibonacci numbers.

당신은 곧 এই 빈약한 대수에 기생하는 카타모피즘 따위인 `cata fib`이, 놀랍게도 그 유명한 피보나치 수열(Fibonacci numbers)을 쳐 계산해 내고 자빠졌다는 사실에 아주 손쉽게 뇌가 굴복하게 될 겁니다.

In general, an algebra for NatF defines a recurrence relation: the value of the current element in terms of the previous element. A catamorphism then evaluates the n-th element of that sequence.

좀 더 거시적인 안목으로 찌껄이자면, 저 `NatF`라는 함자를 호위하는 대수란 건 항상 어떤 점화식(recurrence relation) 하나를 지루하게 규정해 댑니다: 직전의 찌꺼기(previous element) 따위를 긁어모아 현재의 불쌍한 알맹이(current element) 값을 정해버리는 변태적인 짓 말입니다. 그리고 카타모피즘이란 놈은 기어코 그 점화식 수열의 `n`번째 알맹이의 목을 비틀고 평가(evaluates)하여 실체(값)를 뽑아냅니다.

## Folds

## 폴드(Folds)

A list of e is the initial algebra of the following functor:

저 `e` 나부랭이들의 리스트(list of e)란 결국 다음과 같은 구질구질한 함자가 싸지른 한낱 초기 대수(initial algebra)에 불과합니다:

```
data ListF e a = NilF | ConsF e a
```

Indeed, replacing the variable a with the result of recursion, which we’ll call List e, we get:

진실로, 저 변수 `a`쪼가리를 기어이 재귀의 산출물 — 즉 우리가 감히 `List e`라 일컬을 — 로 무자비하게 치환해 버리면 다음과 같은 꼬라지를 맞이합니다:

```
data List e = Nil | Cons e (List e)
```

An algebra for a list functor picks a particular carrier type and defines a function that does pattern matching on the two constructors. Its value for NilF tells us how to evaluate an empty list, and its value for ConsF tells us how to combine the current element with the previously accumulated value.

어떤 리스트 함자 나부랭이를 모시는 대수란, 그저 입맛에 맞는 특정 캐리어 타입 하나를 스윽 노획한 다음 그 두 개의 빈약한 생성자 놈들 위에서 패턴 매칭(pattern matching)이나 갈겨대는 하찮은 함수를 정의할 따름입니다. 여기서 `NilF`를 향해 내린 값이란 그 알맹이 없는 '빈 리스트(empty list)' 따윌 어떻게 능욕(평가)해 줄지 가르쳐 주는 꼴이고, `ConsF`를 향한 처분은 지금 손에 잡힌 생때같은 원소를 이전에 구질구질하게 축적해 둔 값어치(previously accumulated value)와 대체 무슨 수로 얼버무려 섞어댈지(combine) 지시할 뿐입니다.

For instance, here’s an algebra that can be used to calculate the length of a list (the carrier type is Int):

가령 예컨대, 여기 어떤 불운한 리스트의 길이(length) 따위나 묵묵히 계산해 바칠 참으로 한심한 대수(이때 캐리어 타입은 그저 `Int` 나부랭이입니다)가 하나 있습니다:

```
lenAlg :: ListF e Int -> Int
lenAlg (ConsF e n) = n + 1
lenAlg NilF = 0
```

Indeed, the resulting catamorphism cata lenAlg calculates the length of a list. Notice that the evaluator is a combination of (1) a function that takes a list element and an accumulator and returns a new accumulator, and (2) a starting value, here zero. The type of the value and the type of the accumulator are given by the carrier type.

실로 이건 한 치의 거짓 없이, 이 작태가 낳은 카타모피즘 `cata lenAlg`라는 놈은 결국 어떤 잡스러운 리스트의 길이를 꾸역꾸역 계산해 냅니다. 여기서 그 평가자란 놈이 기실 (1) 리스트의 원소 하나랑 그동안 굴려온 누산기(accumulator) 덩어리를 덥석 물고선 기어이 또 다른 새파란 누산기를 뱉어내는 어떤 야비한 함수 하나, 그리고 (2) 맨땅에 헤딩할 때 쓸 시발점(starting value), 즉 여기서의 0이라는 한 쌍의 구질구질한 조합(combination)에 불과하다는 뼈아픈 진실을 똑똑히 눈여겨 보십시오. 저 값의 타입이나 더러운 누산기의 타입 따위는 전부 다 저 비루한 캐리어 타입(carrier type)의 손아귀에 쥐락펴락 결정됩니다.

Compare this to the traditional Haskell definition:

이걸 세상 널려 빠진 전통적인 Haskell의 구린내 나는 정의와 한번 맞대어 보시죠:

```
length = foldr (\e n -> n + 1) 0
```

The two arguments to foldr are exactly the two components of the algebra.

저 불결한 `foldr`의 아가리로 처박히는 두 놈의 인수들은, 소름 돋게도 조금 전 그 대수(algebra)를 구성하던 그 두 마리의 부속품들과 토씨 하나 안 틀리고 정확히(exactly) 똑똑 떨어집니다.

Let’s try another example:

어디 또 다른 잡스런 예제나 한번 건드려 볼까요:

```
sumAlg :: ListF Double Double -> Double
sumAlg (ConsF e s) = e + s
sumAlg NilF = 0.0
```

Again, compare this with:

이딴 것 역시 다음 찌끄러기와 비교질이나 해보시길:

```
sum = foldr (\e s -> e + s) 0.0
```

As you can see, foldr is just a convenient specialization of a catamorphism to lists.

이제 두 눈깔로 똑똑히 조명했듯이, 저 빌어먹을 `foldr` 따위는 기실 고결한 카타모피즘을 고작 '리스트(lists)'라는 변태적인 규격에 맞춰 어설프게 특수화(specialization)시켜 놓은, 그저 몹시도 편의주의적인 사생아에 불과합니다.

## Coalgebras

## 코알지브라 (Coalgebras)

As usual, we have a dual construction of an F-coagebra, where the direction of the morphism is reversed:

늘 하던 뻔한 수작대로, 우린 저 사상의 모가지를 확 비틀어 방향을 반대로 돌려놓은 F-코대수(F-coalgebra)라는 쌍대(dual)의 건축물을 손에 쥘 수 있습니다:

```
a -> F a
```

Coalgebras for a given functor also form a category, with homomorphisms preserving the coalgebraic structure. The terminal object (t, u) in that category is called the terminal (or final) coalgebra. For every other algebra (a, f) there is a unique homomorphism m that makes the following diagram commute:

특정 함자에 종속된 저 코대수(Coalgebras) 찌꺼기들 역시 준동형 사상이 코대수적 구조를 병적으로 보존해 대며 지들끼리 하나의 범주를 형성합니다. 그 빌어먹을 범주 안의 끝 대상(terminal object)인 `(t, u)`를 일컬어 사람들은 끝 코대수(terminal coalgebra), 혹은 최종 코대수(final coalgebra)라 떠받듭니다. 다른 그 어떤 대수(여기서는 코대수) `(a, f)`를 들이밀어도 오직 단 하나뿐인 준동형 사상 `m`이 존재하여, 다음 다이어그램을 닥치고 교환되게 만듭니다:

A terminal colagebra is a fixed point of the functor, in the sense that the morphism u :: t -> F t is an isomorphism (Lambek’s theorem for coalgebras):

끝 코대수란 건 필경 저 사상 `u :: t -> F t` 따위가 또 하나의 동형사상(isomorphism)으로 등극한다는 측면에서, 결국 함자의 고정점(fixed point) 나부랭이에 불과합니다 (이것이 람벡의 정리 코대수 버전입니다):

```
F t ≅ t
```

A terminal coalgebra is usually interpreted in programming as a recipe for generating (possibly infinite) data structures or transition systems.

끝 코대수는 프로그래밍 바닥에서는 흔히, 저주받은 무한대의 데이터 구조들이나 전이 시스템(transition systems)들을 토해내는 기괴한 레시피쯤으로 허세 있게 해석되곤 합니다.

Just like a catamorphism can be used to evaluate an initial algebra, an anamorphism can be used to coevaluate a terminal coalgebra:

카타모피즘(catamorphism) 쪼가리가 초기 대수의 숨통을 끊고 평가자 역할을 해대듯, 아나모피즘(anamorphism)이라는 이름의 변종은 이 끝 코대수 놈을 향해 코평가(coevaluate)라는 역겨운 짓거릴 서슴지 않습니다:

```
ana :: Functor f => (a -> f a) -> a -> Fix f
ana coalg = Fix . fmap (ana coalg) . coalg
```

A canonical example of a coalgebra is based on a functor whose fixed point is an infinite stream of elements of type e. This is the functor:

코대수의 아주 진부하고 판에 박힌(canonical) 예시 하나는, 바로 징그러운 `e` 타입 알맹이들의 무한 스트림(infinite stream) 따위를 자기 고정점 삼아 깔고 앉은 어느 함자 위에 구축됩니다. 그 거지 같은 함자는 이렇습니다:

```
data StreamF e a = StreamF e a
  deriving Functor
```

and this is its fixed point:

그리고 요놈이 바로 그 빌어먹을 고정점입니다:

```
data Stream e = Stream e (Stream e)
```

A coalgebra for StreamF e is a function that takes the seed of type a and produces a pair (StreamF is a fancy name for a pair) consisting of an element and the next seed.

저 `StreamF e` 밑에서 핥아대는 코대수란 건 고작 `a` 타입의 씨앗(seed) 하나를 날름 삼키고선, 어떤 알맹이 하나와 또 다른 다음 번 씨앗(next seed)으로 버무려진 한 쌍의 쓰레기 더미(`StreamF`란 건 그저 '쌍'을 거창하게 포장한 이름일 뿐입니다)를 게워내는 하찮은 함수에 지나지 않습니다.

You can easily generate simple examples of coalgebras that produce infinite sequences, like the list of squares, or reciprocals.

당신은 당장 제곱수들의 리스트나 역수(reciprocals) 따위를 미친 듯이 무한대로 뽑아내는 코대수 예제쯤은 눈 감고도 뚝딱 창조해 낼 수 있을 겁니다.

A more interesting example is a coalgebra that produces a list of primes. The trick is to use an infinite list as a carrier. Our starting seed will be the list [2..]. The next seed will be the tail of this list with all multiples of 2 removed. It’s a list of odd numbers starting with 3. In the next step, we’ll take the tail of this list and remove all multiples of 3, and so on. You might recognize the makings of the sieve of Eratosthenes. This coalgebra is implemented by the following function:

좀 더 흥미가 동하는 악질적인 예시는 기어코 소수(primes)들의 리스트를 게워내는 코대수입니다. 이 교활한 속임수(trick)는 무한 리스트 따위를 캐리어로 노려 써먹는 데 있습니다. 우리가 처음 내던질 씨앗은 `[2..]` 따위의 무식한 리스트입니다. 그다음 타자로 등판할 씨앗은 이 오리지널 리스트의 꼬리를 싹 자른 뒤 2의 배수 나부랭이들을 죄다 학살해 버린 残骸물입니다. 그럼 3부터 시작하는 불쌍한 홀수들의 리스트가 남겠죠. 다음 단계에선 또 이 시궁창 리스트의 꼬리를 잡고는 3의 배수들을 일렬로 세워 모가지 쳐내는 짓거리를 지긋지긋하게 반복합니다. 어디서 본 듯한 그놈의 '에라토스테네스의 체(sieve of Eratosthenes)' 냄새가 진동하지 않습니까? 이 지독한 코대수는 다음의 함수 꼬라지로 구현됩니다:

```
era :: [Int] -> StreamF Int [Int]
era (p : ns) = StreamF p (filter (notdiv p) ns)
    where notdiv p n = n `mod` p /= 0
```

The anamorphism for this coalgebra generates the list of primes:

이 코대수에다 야만적인 아나모피즘을 갈겨버리면 끝내 소수들의 리스트가 창조됩니다:

```
primes = ana era [2..]
```

A stream is an infinite list, so it should be possible to convert it to a Haskell list. To do that, we can use the same functor StreamF to form an algebra, and we can run a catamorphism over it. For instance, this is a catamorphism that converts a stream to a list:

어차피 스트림(stream)이란 건 한낱 역겨운 무한 리스트일 뿐이니, 여하튼 이 더러운 덩어리를 고상한 Haskell 리스트로 세탁하는 게 가능해야만 마땅할 겁니다. 그따위 짓거릴 하려면 우린 예의 그 잘난 `StreamF` 함자를 재활용하여 대수를 빚어내고, 그 위로 가차 없이 카타모피즘을 끌고 지나가게 맹글 수 있습니다. 가령 이거 한 번 보시죠, 스트림을 무식하게 리스트로 우겨 박는 카타모피즘입니다:

```
toListC :: Fix (StreamF e) -> [e]
toListC = cata al
   where al :: StreamF e [e] -> [e]
         al (StreamF e a) = e : a
```

Here, the same fixed point is simultaneously an initial algebra and a terminal coalgebra for the same endofunctor. It’s not always like this, in an arbitrary category. In general, an endofunctor may have many (or no) fixed points. The initial algebra is the so called least fixed point, and the terminal coalgebra is the greatest fixed point. In Haskell, though, both are defined by the same formula, and they coincide.

바로 이 지점에서, 뻔뻔하게도 똑같은 고정점(fixed point) 하나가 한 입으로 두말하듯 동일한 자기 함자의 초기 대수(initial algebra) 노릇과 끝 코대수(terminal coalgebra) 행세를 동시에(simultaneously) 꿰차고 자빠졌습니다. 물론 개나 소나 다 기어다니는 임의의 범주 속에서야 이런 꼴이 늘상 터지진 않습니다. 통상적인 바닥에선, 어떤 미친 자기 함자가 수십 개의 고정점을 줄줄이 싸지르거나 아예 단 하나도 없을 수도 있죠. 그놈의 초기 대수란 건 흔히 말채찍 휘두르듯 '최소 고정점(least fixed point)'이라 불리고, 반대로 끝 코대수란 건 탐욕스럽게도 '최대 고정점(greatest fixed point)'으로 일컬어집니다. 헌데 이 빌어먹을 Haskell의 세계에서는, 이 어이없는 두 마리가 사실상 토씨 하나 안 틀린 같은 공식으로 빚어지는 바람에 소름 돋게도 지들끼리 합체(coincide)해 버립니다.

The anamorphism for lists is called unfold. To create finite lists, the functor is modified to produce a Maybe pair:

리스트에다 내리꽂는 아나모피즘 나부랭이는 흔히 얼간이처럼 `unfold`라 지껄여집니다. 시시콜콜한 유한 리스트를 쥐어짜 내려면, 그 함자는 오만상을 쓰고 `Maybe` 쌍을 낳도록 덜떨어지게 조작(modified)되어야 합니다:

```
unfoldr :: (b -> Maybe (a, b)) -> b -> [a]
```

The value of Nothing will terminate the generation of the list.

이 짓거리에서 저 `Nothing` 찌꺼기 값이 툭 튀어나오는 순간, 끝끝내 리스트의 징그러운 번식이 학살당하고(terminate) 끝장납니다.

An interesting case of a coalgebra is related to lenses. A lens can be represented as a pair of a getter and a setter:

이 코대수의 기묘하고 흥미로운 사례 중 하나는 바로 렌즈(lenses) 나부랭이와 지독하게 얽혀있습니다. 렌즈란 건 그저 값싼 게터(getter)와 세터(setter) 한 쌍으로 조악하게 표출(represented)될 수 있습니다:

```
set :: a -> s -> a
get :: a -> s
```

Here, a is usually some product data type with a field of type s. The getter retrieves the value of that field and the setter replaces this field with a new value. These two functions can be combined into one:

여기서 저 `a`란 놈은 대충 `s` 타입의 잡스런 필드 하나를 아가리에 물고 있는 곱 데이터 타입(product data type)입니다. 게터 놈은 냅다 그 빌어먹을 필드의 값을 훔쳐내고, 세터 놈은 그 자리에다 난데없는 새 값을 우겨넣어 바꿔치웁니다. 우린 이 치졸한 두 함수를 또다시 하나로 짓이겨 합쳐버릴 수 있습니다:

```
a -> (s, s -> a)
```

We can rewrite this function further as:

이 흉물스러운 함수는 다음과 같은 쓰레기로 한 술 더 떠 재작성됩니다:

```
a -> Store s a
```

where we have defined a functor:

여기서 우리는 은근슬쩍 이런 함자 하나를 빚어놓았습니다:

```
data Store s a = Store (s -> a) s
```

Notice that this is not a simple algebraic functor constructed from sums of products. It involves an exponential as.

똑똑히 명심하십시오. 이놈은 그깟 곱들의 합(sums of products) 찌부러기로 얼기설기 짜맞춘 그딴 단순무식한 대수적 함자 나부랭이가 아닙니다. 놀랍게도 이놈은 `a^s`라는 기괴한 지수(exponential) 덩어리를 대놓고 빨아들이고 있습니다.

A lens is a coalgebra for this functor with the carrier type a. We’ve seen before that Store s is also a comonad. It turns out that a well-behaved lens corresponds to a coalgebra that is compatible with the comonad structure. We’ll talk about this in the next section.

렌즈란 결코 만만한 게 아니라, 이 해괴한 함자 밑에서 기생하며 캐리어 타입으로 저 구역질 나는 `a`를 인질로 잡은 또 하나의 코대수에 속합니다. 우린 앞서 저 `Store s` 놈이 기실 끔찍한 코모나드(comonad) 중 하나였음을 두 눈깔로 확인한 바 있죠. 이 바닥의 결론인즉, 아주 번듯하고 말 잘 듣는(well-behaved) 렌즈 나부랭이는 결국 예의 그 코모나드 구조와 비열하게 양립(compatible)해 먹는 어느 코대수와 일맥상통한다는 전말입니다. 이 씹어먹을 소리에 대해선 이 지긋지긋한 다음 챕터에서 아주 골수까지 파헤쳐 떠들어대겠습니다.

## 도전 과제(Challenges)

- Implement the evaluation function for a ring of polynomials of one variable. You can represent a polynomial as a list of coefficients in front of powers of x. For instance, 4x2-1 would be represented as (starting with the zero’th power) [-1, 0, 4].
- Generalize the previous construction to polynomials of many independent variables, like x2y-3y3z.

- 위에서 한 짓거리를 `x^2y - 3y^3z` 따위의 잡다한 독립 변수(independent variables)들이 뒤섞인 다항식으로까지 일반화(generalize)해 보십시오.
- 2×2 정사각 행렬 환의 대수를 구현해 보십시오.
- Define a coalgebra whose anamorphism produces a list of squares of natural numbers.
- Use unfoldr to generate a list of the first n primes.

- 그놈의 아나모피즘(anamorphism)이 자연수들의 제곱수 리스트를 뱉어내게 만드는 코대수를 친히 정의해 보십시오.
- `unfoldr` 따위를 써먹어, 처음부터 `n`번째까지의 소수(primes) 리스트를 쥐어짜 내보십시오.
